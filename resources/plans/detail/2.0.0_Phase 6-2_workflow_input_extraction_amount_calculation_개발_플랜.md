# 2.0.0 Phase 6-2 — Workflow Input Extraction + amount_calculation 개발 플랜

## Context

Phase 6-1 에서 workflow dispatch 인프라와 첫 generic workflow(`date_calculation`) 가 올라갔지만, 사용자의 자연어 질문에서 `start` / `end` 같은 값을 꺼내는 경로가 없어 실사용 답변은 항상 `MISSING_INPUT` (`"계산하려면 start, end 정보가 필요합니다."`) 으로 수렴하는 상태다. 인프라는 검증됐지만 체감 가치가 아직 없다.

Phase 6-2 가 이 gap 을 채운다. 두 덩어리의 작업을 한 PR 에 묶는다:

1. **`workflow_input` 자동 추출** — 각 workflow 가 필요로 하는 필드 스펙을 선언하고, 질문에서 값을 뽑아 `state.workflow_input` 을 채우는 레이어. **regex 우선(싸고 빠름) + 부족하면 LLM (`gpt-4o-mini`) 로 채워넣기** 의 하이브리드. 정형 패턴(날짜, 숫자) 은 regex 한 방에 끝나고, 지시어·돌려 말함은 LLM 이 마무리.
2. **두 번째 generic workflow `amount_calculation`** — 금액/숫자 목록을 받아 합계·평균·차이를 계산. Phase 5 `parse_money / sum_amounts / average_amount` 조합.

Phase 6-2 가 머지되는 순간 `date_calculation` 도 드디어 실질적 답변을 돌려주기 시작한다 (regex 가 `YYYY-MM-DD` 류를 뽑아주기만 하면 OK). `amount_calculation` 은 인프라가 준비되자마자 붙는 두 번째 검증 사례.

### 이전 Phase 와의 관계

- **Phase 4-3 query_rewriter** — 같은 "질문 → cheap LLM → 구조화된 결과" 패턴을 재사용. 프롬프트 외부화 + fallback 실패 처리 철학 동일.
- **Phase 5 core** — `parse_date`, `parse_money`, `ValidationResult`, `WorkflowResult` 모두 그대로 씀.
- **Phase 6-1** — `registry.WorkflowEntry` 가 확장되어 `input_schema` 를 들고 다니게 된다. dispatch 는 변경 없음. workflow_node 에 extraction 한 단계 추가.

## 핵심 설계 결정

### 1. `FieldSpec` — 각 workflow 가 "필요한 값" 을 선언

```python
# chat/workflows/domains/field_spec.py
from dataclasses import dataclass, field
from typing import Mapping

@dataclass(frozen=True)
class FieldSpec:
    type: str                                 # 'date' | 'number' | 'money' | 'enum' | 'text'
    required: bool = True
    aliases: tuple[str, ...] = ()             # 자연어 힌트 ("시작일", "부터", "start")
    default: object | None = None             # required=False 일 때 쓰임
    enum_values: Mapping[str, tuple[str, ...]] = field(default_factory=dict)
    # ↑ enum 타입에만. 키=정규화된 값, 값=매칭에 쓰일 자연어 토큰들.
    # 예) {'days': ('일', '며칠'), 'months': ('개월', '달'), 'years': ('년')}
```

- **Phase 6-2 기준 `type` 4 개만**: `date / number / money / enum`. `text` 는 미리 칸만 두고 구현 생략.
- **정형 타입(`date/number/money`)** 은 regex 단계에서 찾는다. **`enum`** 은 토큰 매칭으로 찾는다. LLM 은 **정형에서 못 찾은 필드에 한해** 나머지를 채운다.

### 3. 추출 흐름 (`workflow_input_extractor.extract`)

```python
def extract(
    question: str,
    history: list[dict],
    schema: Mapping[str, FieldSpec],
) -> tuple[dict[str, Any], Optional[Usage], Optional[str]]:
    """
    반환: (workflow_input_dict, usage, model)
    - usage/model: LLM 이 실제로 호출됐을 때만 채움 → pipeline 이
      record_token_usage 호출할지 결정.
    """
    if not schema:
        return {}, None, None

    # Step 1: regex/토큰 기반 정형 추출
    extracted: dict[str, Any] = {}
    extracted.update(_extract_dates(question, schema))
    extracted.update(_extract_numbers(question, schema))
    extracted.update(_extract_enums(question, schema))

    # Step 2: required 중 아직 비어있는 것이 있으면 LLM 보조
    missing_required = [
        name for name, spec in schema.items()
        if spec.required and name not in extracted
    ]
    if not missing_required:
        return extracted, None, None

    try:
        llm_output, usage, model = _call_llm_extractor(
            question, history, schema, already_extracted=extracted,
        )
        extracted.update(llm_output)
        return extracted, usage, model
    except Exception as exc:
        logger.warning('workflow_input LLM 추출 실패, regex 결과만 사용: %s', exc)
        return extracted, None, None
```

- **regex 가 먼저 달리는 이유**: 대부분의 날짜/숫자 질문은 `2025-01-01`, `3개월`, `1000원` 형태라 LLM 호출 없이 끝나면 지연·비용 절약.
- **LLM fallback 실패 시**: regex 결과만 들고 workflow 로 넘어감 → workflow 의 `validate()` 가 MISSING_INPUT 으로 "무엇이 아직 필요한지" 사용자에게 안내.
- **history 사용**: "이번 달" 같이 앞서 언급된 날짜에 의존하는 질문을 LLM 이 해석할 수 있도록 최근 3~4 턴 전달 (query_rewriter 와 동일 전략).

### 4. Regex·토큰 추출 세부

```python
# _extract_dates
_DATE_RE = re.compile(
    r'(?:\d{2,4}[-./]\d{1,2}[-./]\d{1,2})'
    r'|(?:\d{2,4}년\s*\d{1,2}월\s*\d{1,2}일)'
)
# 질문에서 나오는 순서대로 최대 2 개를 start / end 에 배정.
# date 타입 필드가 2개 이상이면 schema 선언 순서 기준.
```

```python
# _extract_numbers / _extract_money
_MONEY_RE = re.compile(r'(?:[\d,]+)\s*(?:원)')
_INT_RE   = re.compile(r'(?<!\d)[-+]?\d[\d,]*(?!\d)')
# money 필드가 우선 _MONEY_RE 를, 일반 number 필드는 _INT_RE 를 쓰되
# 이미 money 로 매치된 구간은 제외.
```

```python
# _extract_enums
# FieldSpec.enum_values 를 iterate — 가장 먼저 매치된 토큰의 정규화 값 사용.
```

**edge case**:
- 숫자가 너무 많으면(예: 5 개 이상) `amount_calculation` 입장에서 합계/평균 후보로 전부 수집 → 첫 5개까지만 인정.
- 날짜 2 개만 필요하지만 3 개가 찍혀 있으면 첫 2개만 take.
- regex 가 금액/일반 숫자 양쪽에 매치되는 경우 → `money` 필드가 우선 권리.

기대 출력:

### 6. `workflow_node` 통합

```python
def workflow_node(state):
    key = (state.get('workflow_key') or '').strip()
    if not key or not registry.has(key):
        return single_shot_node(state)

    entry = registry.get(key)
    extracted, usage, model = extract(
        question=state['question'],
        history=state.get('history', []),
        schema=entry.input_schema,
    )
    if usage and model:
        record_token_usage(model, usage)

    result = dispatch.run(key, extracted)
    reply = build_reply_from_result(result, workflow_key=key)
    return {
        'result': QueryResult(reply=reply, sources=[], total_tokens=0, chat_log_id=None),
    }
```

- `state.workflow_input` 필드는 유지하되 외부 지정이 없을 때 extractor 결과로 덮어 쓰는 것으로 의미 정돈. (단위 테스트가 임의 주입할 때 여전히 활용 가능 → `state.workflow_input` 가 있으면 그걸 우선 사용하도록 guard 한 줄 추가.)
- TokenUsage 기록은 `query_rewriter` 와 같은 방식.

### 7. `date_calculation` 의 `input_schema`

```python
input_schema = {
    'start': FieldSpec(type='date', required=True, aliases=('start', '시작', '부터')),
    'end':   FieldSpec(type='date', required=True, aliases=('end', '종료', '끝', '까지')),
    'unit':  FieldSpec(
        type='enum', required=False, default='days',
        aliases=('unit', '단위'),
        enum_values={
            'days':   ('일', '며칠', 'days'),
            'months': ('개월', '달', 'months'),
            'years':  ('년', 'years'),
        },
    ),
}
```

aliases 는 LLM 프롬프트에 주는 힌트로만 쓰인다 (regex 단계에서는 사용 X). alias 를 정확하게 쓴다고 regex 가 더 잘 찾지는 않음.

### 8. `amount_calculation` 명세

입력 스펙:
- `values: list[int]` — 정규화된 금액/수치 (regex `_MONEY_RE` → fallback `_INT_RE`).
- `op: enum` — `'sum' | 'average' | 'diff'` (기본 `'sum'`).

실행:
- `values` 가 2 개 미만 + `op` 가 `diff` → `invalid_input(['diff 연산은 값이 2 개 이상 필요합니다.'])`.
- `op=='sum'` → `sum_amounts`.
- `op=='average'` → `average_amount`.
- `op=='diff'` → `max(values) - min(values)` (두 값 이상일 때 최고·최저 차).

입력 스키마:
```python
input_schema = {
    'values': FieldSpec(type='number_list', required=True, aliases=('values', '금액', '숫자')),
    'op':     FieldSpec(
        type='enum', required=False, default='sum',
        aliases=('op', '연산', '방식'),
        enum_values={
            'sum':     ('합계', '합', 'total', '전체', '더해'),
            'average': ('평균', 'average', 'avg'),
            'diff':    ('차이', 'difference', '차'),
        },
    ),
}
```

> **참고**: `type='number_list'` 는 regex 추출기가 여러 숫자를 모아 list 로 돌려주도록 하는 별도 분기. FieldSpec 의 `type` 리터럴에 추가.
