# 2.0.0 Phase 7-2 — Agent Graph Wiring 개발 플랜

## Context

Phase 7-1 머지 직후 상태:

- `chat/services/agent/` runtime 본체 + 단위 테스트 286/286 green.
- 진입점은 `run_agent(question, history) -> WorkflowResult` 로 안정화. 종료 사유는 `OK / NOT_FOUND / UPSTREAM_ERROR` 중 하나로 변환된다 (`UNSUPPORTED` 는 라우터 책임이라 agent 가 만들지 않음).
- **하지만** `chat/graph/app.py:46` 의 `ROUTE_AGENT` 매핑은 여전히 single_shot 폴백. 즉 agent route 로 매치된 질문도 사용자에게 가는 응답은 기존 single_shot 과 동일.
- agent 결과를 사용자 reply 문자열로 변환하는 포맷터도 아직 없다.

Phase 7-2 의 목표는 **7-1 의 runtime 을 graph 와 view 까지 끝까지 잇는 것**. 새 도구·새 기능·새 도메인을 추가하지 않는다 — 결선·reply·smoke 만 한다.

회귀 0 약속: 7-2 머지 시점에 single_shot / workflow 경로 동작은 변하지 않는다. agent route 로 매치된 질문만 새 경로로 흐르며, 그것마저 fatal 시점엔 사용자에게 친절한 한국어 카피를 보여준다.

## 핵심 설계 결정

### 1. agent_node 는 workflow_node 의 거울

`chat/graph/nodes/workflow.py` 가 이미 잘 정착된 패턴 — 같은 모양으로 따라간다.

```python
# chat/graph/nodes/agent.py (요지)
def agent_node(state: GraphState) -> dict:
    raw_question = state.get('question') or ''
    history = state.get('history') or []

    effective_question = raw_question
    if history:
        effective_question, rw_usage, rw_model = rewrite_query_with_history(
            raw_question, history,
        )
        if rw_usage and rw_model:
            record_token_usage(rw_model, rw_usage)

    result = run_agent(effective_question, history=history)
    reply = build_reply_from_agent_result(result)

    logger.info(
        'agent 실행: status=%s value=%r',
        result.status.value,
        result.value,
    )

    return {
        'result': QueryResult(
            reply=reply,
            sources=[],
            total_tokens=0,
            chat_log_id=None,
        ),
    }
```

**왜 history 가 있을 때만 rewriter** — `rewrite_query_with_history` 자체가 history 가 빈 경우 LLM 호출 없이 원본을 반환하도록 만들어졌지만, 여기선 명시적 `if history:` 분기로 의도를 가시화. 코드 리뷰·dev log·로그 추적이 쉬워진다.

**rewriter 실패 처리** — 내부적으로 LLM 실패 시 원본을 반환하므로 graph 단에선 별도 `try/except` 불필요. (Phase 4-3 결정 그대로.)

**`state.question` 은 그대로** — view·logger·history 는 raw 질문이 보여야 자연스럽다 (사용자가 입력한 그대로). rewriter 결과는 `effective_question` 지역 변수로만 흘려 `run_agent` 의 첫 인자로 넘긴다. 7-1 의 `AgentState.question` 옵션 (a) 와 일관 — `AgentState` 자체에 `search_query` 같은 별도 필드를 추가하지 않는다 (YAGNI).

**TokenUsage 이중 기록 방지** — `run_agent` 안에서 매 LLM 호출마다 이미 `record_token_usage` 호출 중. agent_node 는 rewriter 호출분만 기록 → 한 턴 최대 LLM 호출 ≈ rewriter 1 + agent step (max_iterations+1) ≈ 6 회.

### 2. reply 포맷터는 `chat/services/agent/reply.py` 로 분리

Phase 6-3 의 `chat/workflows/domains/reply.py` 와 합치지 않는다.

이유:

- workflow reply 는 `_ok_formatters` 가 `workflow_key` 별로 분기 (date_calculation / amount_calculation / table_lookup). agent 는 `workflow_key` 가 없다. 한 모듈에 끼워 넣으면 (a) `workflow_key='agent'` hack 또는 (b) `if workflow_key is None:` 분기가 늘어남.
- agent 가 만들 수 있는 status 가 세 개로 좁고 (`OK / NOT_FOUND / UPSTREAM_ERROR`), workflow 의 여섯 status 와 분기 형태가 다르다.
- 별도 모듈로 두면 dev log 의 "agent 가 `WorkflowResult` 를 빌려쓴다는 의도된 어색함" 도 선명해진다.

```python
# chat/services/agent/reply.py (요지)
def build_reply_from_agent_result(result: WorkflowResult) -> str:
    status = result.status

    if status == WorkflowStatus.OK:
        # agent 의 final_answer 는 이미 한국어 응답. value 그대로 노출.
        return str(result.value)

    reason = (result.details or {}).get('reason', '')

    if status == WorkflowStatus.NOT_FOUND:
        return reason or '요청에 맞는 자료를 찾지 못했습니다. 관련 문서가 업로드되어 있는지 확인해 주세요.'

    if status == WorkflowStatus.UPSTREAM_ERROR:
        return reason or '일시적인 오류로 이번 요청을 처리하지 못했습니다. 잠시 후 다시 시도해 주세요.'

    # MISSING_INPUT / INVALID_INPUT / UNSUPPORTED 는 agent 가 만들지 않는다.
    raise ValueError(f'agent 가 만들면 안 되는 status: {status!r}')
```

이 명시적 ValueError 가 회귀 가드 — agent runtime 이 잘못된 status 를 만들기 시작하면 단위 테스트가 즉시 깨진다.

### 3. graph wiring 한 줄 교체

```python
# chat/graph/app.py
builder.add_node('agent', agent_node)             # ← 신규
builder.add_conditional_edges('router', ..., {
    ROUTE_SINGLE_SHOT: 'single_shot',
    ROUTE_WORKFLOW: 'workflow',
    ROUTE_AGENT: 'agent',                         # ← single_shot 폴백 → 신규 노드
})
builder.add_edge('agent', END)                    # ← 신규
```

헤더 docstring 의 graph shape 도 동시 갱신:

```
... agent → END   (Phase 7-2: agent_node 가 run_agent 를 실행)
```

### 5. 회귀 가드

- single_shot / workflow 경로 동작 변경 0 — 기존 단위 테스트가 한 건도 깨지지 않아야 한다.
- agent 가 절대 만들지 않아야 하는 status (`MISSING_INPUT / INVALID_INPUT / UNSUPPORTED`) 가 reply 단에서 ValueError 로 잡힌다 — `test_agent_reply.py` 의 명시적 가드 (§5-1 정책).
- **graph 결선 자체** (`add_node` / conditional edge / `add_edge` / ROUTE_AGENT 매치 시 agent_node 호출 / 미매치 시 호출 0) — `test_graph_agent_wiring.py` 가 `_compiled_graph.cache_clear()` + mock 노드로 `run_chat_graph(...)` 를 한 번 돌려 잡는다. 7-1 머지 시점까진 `chat.tests` 어디에서도 `run_chat_graph` 를 호출하지 않았으므로 본 PR 이 **graph 단 자동 회귀 테스트의 첫 사례**가 된다.

### 5-1. reply ValueError 정책 (P3 명시)

`chat/services/agent/reply.py` 가 `MISSING_INPUT / INVALID_INPUT / UNSUPPORTED` 에 도달하면 `ValueError` 를 raise → Django 500. 이건 **fail-fast invariant guard** 로 의도된 정책이다 — 사용자에게 보여줄 친절한 카피 약속과 모순되지 않는다는 게 핵심:

- agent runtime 이 실제로 fatal 에 도달하는 경우 (`MAX_ITERATIONS_EXCEEDED / FATAL_ERROR / NO_MORE_USEFUL_TOOLS / INSUFFICIENT_EVIDENCE`) 는 7-1 의 `to_workflow_result` 가 이미 `OK / NOT_FOUND / UPSTREAM_ERROR` 셋 중 하나로 변환. 이 셋은 reply 포맷터가 한국어 카피로 끝까지 처리한다 → 사용자 약속 유지.
- `MISSING_INPUT / INVALID_INPUT / UNSUPPORTED` 는 **agent runtime 이 만들 수 없는 코드 경로**다. `chat/tests/test_agent_result.py` 의 회귀 가드(`UNSUPPORTED` 미생산)와 7-2 의 `test_agent_reply.py` 가 이중으로 봉쇄. 운영에 도달하면 그건 runtime 의 buggy regression 이고, 친절한 fallback 으로 숨기면 그 버그가 묵인된다.
- 따라서 ValueError 는 "운영에 도달하면 안 되는 invariant 위반" 시그널 — 친절히 잡지 않는 게 의도. 친절히 잡으면 dead code 보호용 try/except 가 늘어나고, agent runtime 이 status 어휘를 잘못 확장해도 가시화되지 않는다.

대안 (이 PR 채택 안 함):

- `agent_node` 에서 `try: build_reply_from_agent_result(...) except ValueError: ... UPSTREAM_ERROR fallback` — 운영 안정성은 +α 지만 invariant guard 는 무력화. 향후 status 어휘가 늘면(예: 회사 전용 도메인 추가 시) reply 분기를 손대기 전에 agent_node 의 try/except 가 먼저 적용돼 reply 변환이 silent skip 될 위험.
- `QueryPipelineError` 로 변환 → view 의 502 경로 — 같은 이유로 미채택. 502 도 dead-code 경로면 가시화가 약해진다.

후속 Phase 에서 status 어휘가 실제로 확장되면 (예: `BaseResult` 추출 + `AgentResult` 분리) 이 정책을 재검토.

### 6. 사용자-facing smoke (Phase 7 의 마지막 검증)

7-1 runtime 의 status 매핑 사실 정리 (smoke 기대값 작성에 직결):

- **OK** ← `final_answer` 가 비어있지 않음. LLM 이 "자료를 찾지 못했습니다" 라고 답해도 `value` 가 그 문자열이고 status 는 OK. reply 도 그 문자열을 그대로 노출.
- **NOT_FOUND** ← `final_answer` 의 answer 가 빈 문자열 / 도구 연속 실패 ≥ 3 / 동일 호출 ≥ 3.
- **UPSTREAM_ERROR** ← `max_iterations` 도달 / LLM 호출 자체 실패 / JSON 파싱 두 번 연속 실패.

따라서 smoke 시나리오:

- BO `/bo/router-rules/` 에 `route='agent'` 룰 1~2 개 등록 (예: 키워드 "비교", "더 좋은", "차이점").
- 채팅 UI 에서 다음 시나리오 통과:
  - **agent 매치 + 자료 충분** → 두 자료 비교형 질문에서 ReAct 가 도구를 1~3 회 호출하고 한국어로 답변. 서버 로그: `agent 실행: status=ok ...`.
  - **agent 매치 + LLM 정직 응답** → 자료가 모자란 비교형 질문에서 LLM 이 final_answer 로 "자료를 찾지 못했습니다" 류 한국어를 직접 작성. status=OK, reply 는 그 문자열 그대로 (NOT_FOUND 카피가 아님). dev log §검증에 "이게 7-1 mapping 의 의도된 동작" 한 줄 명시.
  - **agent 매치 + non-agent 질문 회귀** → 같은 세션에서 일반 질문 (single_shot / workflow 매치) 던졌을 때 7-1 머지 직후와 답변 동일.
  - **agent 매치 + fatal** → OPENAI_API_KEY 무효화 후 agent route 매치 → status=UPSTREAM_ERROR → 친절한 한국어 카피 (`'일시적인 오류로...'`).
  - **NOT_FOUND smoke 는 운영에서 재현 어려움** — 도구를 일부러 깨뜨려야 하므로 이 시나리오는 단위 테스트 (`test_agent_react.py` 의 `test_repeated_same_call_terminates_with_not_found` / `test_consecutive_tool_failures_trigger_no_more_useful_tools`) 로만 보장. dev log 에 그 사실 명시.
- 결과를 dev log §검증 절에 기록 — 응답 시간 / LLM 호출 횟수 / 도구 사용 패턴 / 캡처 1~2 장.

### 7. 결과 타입의 어색함은 그대로 유지

7-1 에서 의도한 "agent 가 `WorkflowResult` 를 반환한다" 어색함은 7-2 에서도 유지. reply 분기를 status 만 보고 처리하는 단순함이 변경 비용을 압도하기 때문 — 후속 `BaseResult` 추출 리팩터는 별건. dev log 에 다시 한 번 명시.
