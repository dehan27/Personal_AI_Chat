# 2.0.0 Phase 6-3 — table_lookup + 남은 정책 다듬기 개발 플랜

## Context

Phase 6-1 이 dispatch 인프라 + `date_calculation` 을, Phase 6-2 가 `workflow_input_extractor` + `amount_calculation` 을 올려 자연어 질문이 처음으로 workflow 경로까지 도달했다. 이제 Phase 6 의 세 번째이자 마지막 generic workflow 인 **`table_lookup`** 을 추가한다.

`table_lookup` 은 앞선 두 workflow 와 성격이 조금 다르다:
- **순수 계산기가 아님** — 어떤 값을 찾을지 자체를 사용자 질문이 정한다("표에서 A항목 금액", "3행 2열 값").
- **회사 자료에 의존** — 표 자체가 업로드된 문서 청크에 있다. Phase 5 retrieval (`files.services.retriever.search_chunks`) 이 필요.
- **LLM 보조가 정당화된다** — 표 안의 행/열 명이 사용자의 표현과 다르게 적혀 있어도 해석해야 함. 그럼에도 출력은 **결정적 셀 값 하나**(문자열) 라서 agent 가 아니라 workflow 자리.

같은 PR 에서 Phase 6 설계 §5-3, §7-3 의 `unsupported` 정책도 다듬는다. 현재 reply.py 의 UNSUPPORTED 메시지는 "현재 지원하지 않는 작업입니다." 정도의 축약이라 **원인 + 대안 제시** 가 약하다.

---

## Scope

**포함**
- `FieldSpec` 에 `'text'` 타입 추가 — 자유형 문자열 필드(규모 제한 없이 질문 원문을 받는 데 사용).
- `workflow_input_extractor` 에 text 처리 경로 추가 — regex 단계에서 못 채워진 `text` 필드는 **질문 원문을 그대로 채움** (table_lookup 의 `query` 가 이를 씀).
- `workflow_node` 에 **history-aware query rewrite** 통합 — schema 에 `text` 타입 필드가 있는 workflow 에 한해 `chat.services.query_rewriter.rewrite_query_with_history` 를 먼저 한 번 돌려 extractor·workflow 전체가 rewritten 질문을 보게 한다. `"그 표에서 제일 큰 금액"` 같은 지시어 의존 후속 질문이 single_shot 과 동일한 방식으로 맥락화되도록.
- **`WorkflowStatus` 확장** — 현재 `UNSUPPORTED` 가 "지원 안 함" · "데이터 없음" · "실행 실패" 세 가지를 섞고 있어 reply·관측·향후 재시도 정책에서 구분이 안 됨. Phase 6-3 에서 아래 두 status 와 그에 대응하는 `WorkflowResult.not_found(...)` / `WorkflowResult.upstream_error(...)` 팩토리를 추가:
  - `NOT_FOUND`     — workflow 가 정상 실행됐지만 질문에 맞는 데이터를 못 찾음.
  - `UPSTREAM_ERROR` — LLM / 네트워크 / 파서 등 일시적 실행 실패.
- 기존 `UNSUPPORTED` 는 원 의미("이 workflow 는 본래 이 요청을 다루지 않음 — 미등록 key, 스펙 밖 op 등") 로 환원.
- `reply.py` 가 세 상태를 각기 다른 문구로 포맷 (사용자 안내·다음 행동 달라짐).
- `chat/workflows/core/tables.py` 신규 — 마크다운 표를 `list[dict]` 로 파싱하는 순수 helper. 도메인 로직 없음.
- `chat/workflows/domains/general/table_lookup.py` 신규:
  - `input_schema = {'query': FieldSpec(type='text', required=True)}`
  - `execute`: retrieval → 마크다운 표 청크 필터 → LLM 에게 "cell 값 하나" 프롬프트 → `WorkflowResult.ok(value=<str>, details={'source_document': ..., 'matched_row': ..., 'matched_column': ...})`
  - 후보 표 0건 → `WorkflowResult.not_found('질문에 맞는 표를 찾지 못했습니다. ...')` (자료가 없는 경우).
  - LLM 이 빈 `{}` 또는 `answer` 미포함 응답 → `WorkflowResult.not_found('표에서 해당 값을 찾지 못했습니다. ...')` (표는 있지만 셀 매치 실패).
  - LLM 호출 예외 / JSON 파싱 실패 → `WorkflowResult.upstream_error('표 해석 중 일시적인 오류가 발생했습니다. ...')` (재시도 대상).
  - `UNSUPPORTED` 는 이 workflow 에서는 사용하지 않음 — "이 workflow 범위 밖" 에 해당하는 케이스가 없기 때문 (미등록 key 는 dispatch 단에서 이미 처리).
- `assets/prompts/chat/table_lookup.md` 신규 — 표 해석용 시스템 프롬프트 (Phase 1 `prompt_registry` 등록 필수).
- `reply.py` 에 `table_lookup` 전용 OK 포맷터 추가 + `NOT_FOUND` / `UPSTREAM_ERROR` / `UNSUPPORTED` 세 status 별 기본 문구 분기 (reason pass-through 우선).
- 단위 테스트 — tables.py / table_lookup (mocked retrieval + mocked LLM) / reply formatter 추가분.
- README + dev log + Phase 6 완료 노트.

**제외 (Phase 6-4 / Phase 7 이후)**
- `document_compare` / `conditional_reasoning` / `multi_source_summary` 구현.
- Excel 파일 직접 파싱 (현재는 extraction 단계에서 markdown 으로 이미 변환됨 — 그 결과를 소비만 한다).

## 핵심 설계 결정

### 1. `'text'` FieldSpec 타입의 의미

```python
SUPPORTED_TYPES = ('date', 'number', 'money', 'enum', 'number_list', 'text')
```

- **regex 단계**에서는 아무것도 안 한다 — text 는 "질문 안의 어떤 부분" 이 아니라 "질문 그 자체" 를 담는 용도가 기본이다.
- **extractor 마무리 단계**에서 "text 필드가 아직 비어있고 required 면 `question` 문자열을 그대로 채운다". 이로써 `table_lookup` 이 extractor 한 번만 돌려도 `query` 에 질문이 들어간다.
- LLM fallback 도 text 필드에 한해 스킵 — 이미 질문 자체가 주입되므로 추가 호출 불필요.

대안으로 검토했다가 제외한 것: "schema 에 text 필드가 있으면 workflow_node 가 알아서 `state.question` 을 주입". 너무 magic 하고, 여러 text 필드 시나리오 지원이 어려움.

### 2. 마크다운 표 파서 (`chat/workflows/core/tables.py`)

```python
def parse_markdown_tables(text: str) -> list[dict]:
    """
    text 안의 모든 GFM 스타일 표를 파싱.
    반환: [{'headers': ['A', 'B'], 'rows': [{'A': 'x', 'B': 'y'}, ...]}, ...]
    표 구분자(| --- | --- |) 없는 문단은 무시.
    """
```

- **순수 함수** — Phase 5 core 의 의존 방향 원칙 유지.
- **규모 제한**: 한 청크 안에서 최대 10 개 표 / 각 표 최대 200 행 (안전판).
- **파싱 실패는 빈 리스트** — LLM 이 적당히 해석해 줄 거라고 가정하지 않는다.

파싱 결과는 LLM 프롬프트에 **다시 마크다운으로 직렬화해 넣는다** — LLM 이 마크다운 표를 읽는 능력은 좋음. list[dict] 는 우리가 내부 필터링·로그에 쓰기 좋은 구조라서 중간 산물로만 보유.

### 3. `table_lookup.execute` 흐름

```python
def execute(self, normalized):
    query = normalized['query']

    # Phase 6-3: 원시 search_chunks 가 아니라 single_shot 의 reranker 포함 파이프라인을
    # 그대로 재사용한다. LLM 이 candidate 표 중에서 셀을 고르는 구조상 후보 품질이
    # 답의 질을 직접 좌우하기 때문.
    hits = retrieve_documents(query)

    candidates: list[tuple[ChunkHit, list[dict]]] = []
    for hit in hits:
        tables = parse_markdown_tables(hit.content)
        if tables:
            candidates.append((hit, tables))

    if not candidates:
        return WorkflowResult.not_found(
            '질문에 맞는 표를 찾지 못했습니다. 관련 문서가 업로드되어 있는지 확인해 주세요.'
        )

    try:
        answer, meta = _ask_llm_for_cell(query, candidates)
    except Exception as exc:                                      # noqa: BLE001
        logger.warning('table_lookup LLM 실패: %s', exc)
        return WorkflowResult.upstream_error(
            '표 해석 중 일시적인 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.'
        )

    if not answer:
        return WorkflowResult.not_found(
            '표에서 해당 값을 찾지 못했습니다. 항목 이름을 확인해 주세요.'
        )

    return WorkflowResult.ok(
        value=answer,
        details={
            'source_document': meta.get('source_document'),
            'matched_row': meta.get('matched_row'),
            'matched_column': meta.get('matched_column'),
        },
    )
```

- `retrieve_documents` 는 `chat.services.single_shot.retrieval.retrieve_documents` 로 현재 single_shot 에서 쓰이는 **하이브리드 검색 + LLM rerank** 엔트리. workflow 가 이걸 import 해 씀 → 후보 품질·top_k 상수가 single_shot 과 자동 일치. 추후 "retrieval 이 single_shot 패키지 안에 있는 게 맞나" 를 따지는 리팩터는 별건.
- Status 분리 반영:
  - **후보 표 0건**: `NOT_FOUND` — workflow 는 정상이지만 자료가 없음.
  - **LLM 예외**: `UPSTREAM_ERROR` — 재시도 대상. dev log 에 재시도 정책은 out-of-scope 라 명시.
  - **LLM 이 빈 `{}` 반환(표에서 못 골라냄)**: `NOT_FOUND` — "자료는 있는데 이 질문에 맞는 값이 없음".
- `details['source_document']` — reply 단계에서 "(출처: ...)" 로 인용.
- `TokenUsage` 는 LLM 호출 후 workflow 내부에서 `record_token_usage` 직접 호출. 다른 workflow(date/amount)는 LLM 호출 없음 → workflow_node 쪽 단일 기록 패턴 유지되지만, table_lookup 의 LLM 은 workflow 고유 책임.

### 4. LLM 프롬프트 (`assets/prompts/chat/table_lookup.md`)

입력 구조 (user message):

```
Question: {user_query}

Tables (markdown):
=== Document: {filename_1} ===
{markdown_table_1}

=== Document: {filename_2} ===
{markdown_table_2}

Return JSON only:
```

기대 출력:

```json
{
  "answer": "1,234,567원",
  "source_document": "경조사_규정.pdf",
  "matched_row": "본인 상",
  "matched_column": "경조금"
}
```

- system: "표에서 사용자가 찾는 셀을 딱 하나 골라 JSON 으로만 돌려주세요. 근거 없이 추정하지 마세요. 찾을 수 없으면 `{}` 를 반환."
- parsing 결과 매핑:
  - `{}` 또는 `answer` 키만 비어있음 → **`WorkflowResult.not_found`** (자료에서 매치 실패 — 정상 경로).
  - JSON 파싱 실패 / 예상치 못한 구조 / 호출 예외 → **`WorkflowResult.upstream_error`** (모델/네트워크/포맷 문제 — 재시도 대상).
  - UNSUPPORTED 는 이 단계에서 쓰지 않음. table_lookup 이 "다루지 않는 요청" 인 상황은 dispatch 가 미등록 key 에 대해서만 생성.
- 모델: 기본 `OPENAI_MODEL`(`gpt-4o-mini`). `query_rewriter` / extractor 와 동일 cheap model.

### 5. Reply 포맷터

```python
def _ok_table_lookup(result):
    details = result.details or {}
    row = details.get('matched_row')
    col = details.get('matched_column')
    src = details.get('source_document')
    body = f'{result.value}'
    if row and col:
        body = f'{row} · {col}: {result.value}'
    elif row:
        body = f'{row}: {result.value}'
    if src:
        body = f'{body}\n\n(출처: {src})'
    return body
```

### 6. History-aware query rewrite 재사용

Phase 4-3 `query_rewriter.rewrite_query_with_history` 는 retrieval 앞단에 자립적 검색어를 만들어주는 일을 하는데, workflow 경로 중 retrieval 을 돌리는 workflow(현재 `table_lookup`) 도 동일한 regression 에 노출된다. 예:

```
user: 경조사 규정 알려줘
bot : (경조사 표를 포함한 설명)
user: 그 표에서 제일 큰 금액         ← "그 표" = 맥락 의존
```

이 경우 `search_chunks("그 표에서 제일 큰 금액")` 은 엉뚱한 문서를 긁어올 가능성이 크다.

**대응**: `workflow_node` 가 `registry.get(key).input_schema` 를 검사해 `'text'` 타입 필드가 있는 workflow 에 한해 dispatch 직전 rewrite 를 실행한다. single_shot 과 동일한 규약:

- history 가 비어있으면 rewrite skip (LLM 호출 0).
- rewriter 가 `NOOP` 을 돌려주면 원본 유지.
- rewriter 예외·빈 응답 시 원본으로 폴백.
- rewrite LLM 호출이 실제로 돌았으면 `record_token_usage` 로 기록.

```python
# workflow_node 에서 dispatch 직전
rewritten_question = state.get('question') or ''
schema = entry.input_schema
if _schema_needs_retrieval(schema) and state.get('history'):
    rewritten_question, usage, model = rewrite_query_with_history(
        rewritten_question, state['history'],
    )
    if usage and model:
        record_token_usage(model, usage)

workflow_input, ex_usage, ex_model = extract(
    question=rewritten_question,
    history=state['history'] or [],
    schema=schema,
)
```

`_schema_needs_retrieval(schema)` 는 임시 헬퍼로 `any(spec.type == 'text' for spec in schema.values())`. 추후 workflow 가 더 늘어나 flag 가 필요해지면 `WorkflowEntry.needs_retrieval: bool` 로 공식화 (Phase 6-3 out-of-scope).

단위 테스트:
- rewriter 가 호출되는 workflow(table_lookup) vs 안 되는 workflow(date_calculation) 구분.
- rewriter 가 NOOP 반환 시 원본 질문이 extractor 에 들어감.
- rewriter 실패 시 rewrite 없이 진행.

### 7. Reply 처리 — 세 status 를 각기 다른 톤으로

Phase 6-1 의 reply.py 는 UNSUPPORTED 하나만 다뤘고 "현재 지원하지 않는 작업입니다" 정도로 얕았다. 6-3 의 status 확장에 맞춰 세 갈래를 나눈다:

| status | 기본 문구 (reason 없을 때) | 의미 / 다음 행동 |
|---|---|---|
| `UNSUPPORTED` | "이 질문은 지원하는 workflow 에 해당하지 않습니다. 다른 방식으로 물어봐 주세요." | 요청 유형이 아예 범위 밖 — BO 운영자가 rule 을 재검토 |
| `NOT_FOUND` | "요청에 맞는 자료를 찾지 못했습니다. 관련 문서가 업로드되어 있는지 확인해 주세요." | workflow 자체는 정상 — 자료 추가·검색어 수정 유도 |
| `UPSTREAM_ERROR` | "일시적인 오류로 이번 요청을 처리하지 못했습니다. 잠시 후 다시 시도해 주세요." | 재시도 대상 — 향후 자동 재시도·모니터링 후크 자리 |

공통 규칙:
- workflow 가 `details['reason']` 에 구체 문구를 넣어둔 경우 reply 는 **그대로 pass-through**. "친절함의 책임은 workflow 가 지고, reply 는 포맷만 담당" 이라는 원칙을 Phase 6-3 dev log 에 명문화.
- reason 미지정 시에만 위 기본 문구를 사용.

결과적으로 사용자 시점에서 "우리가 지원 안 한다" / "지원하지만 못 찾았다" / "일시 장애" 가 뚜렷이 구별된다.
