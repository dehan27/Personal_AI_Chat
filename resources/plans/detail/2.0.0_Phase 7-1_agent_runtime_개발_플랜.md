# 2.0.0 Phase 7-1 — Agent Runtime (foundation) 개발 플랜

## Context

Phase 6 가 정형 절차형 질문(`single_shot` / `workflow`)의 두 줄기를 완성했지만, "두 자료 비교", "여러 단계 탐색", "조건 분기 많은 비교 추천" 같이 **한 번의 retrieval 이나 한 workflow 로 끝나지 않는 질문** 은 여전히 single_shot 으로 흐르거나 workflow 의 잘못된 매칭으로 어색한 답에 수렴한다.

Phase 7 의 목표는 이런 multi-hop / 탐색형 질문을 처리할 수 있는 **generic agent (ReAct) 경로** 를 여는 것 — 단, 새 회사 업무 기능이 아니라 **이미 있는 workflow / retrieval 을 도구처럼 재사용** 하는 상위 오케스트레이터.

설계 문서(`resources/plans/2.0.0_Phase 7_Agent_개발_설계.md`) 의 Step 1~6 을 한 번에 묶으면 리뷰·검증 부담이 커지므로 두 단계로 쪼갠다:

- **Phase 7-1 (이 플랜)** — agent runtime 본체. AgentState · Tool 레이어 · ReAct loop · 결과 타입 · 단위 테스트. **graph 와 연결하지 않는다** — `ROUTE_AGENT` 는 여전히 single_shot 으로 폴백 (Phase 4-1 동작 그대로). runtime 은 직접 호출로 검증 가능.
- **Phase 7-2** — graph `agent_node` 연결 · reply 포맷터 · 사용자-facing smoke · BO RouterRule 의 agent route 활성화 · README/dev log.

회귀 0 약속: 7-1 머지 시점에 **서비스 동작은 변하지 않는다**. agent 코드는 import 가능하되 어떤 graph 경로도 그것을 호출하지 않는다.

## Scope

**포함**
- `chat/services/agent/` 패키지 신설.
- `chat/services/agent/state.py` — `AgentState` dataclass. 7-1 에서 `question` 은 **사용자 원본 입력 그대로** (history-aware rewrite 는 Phase 7-2 의 graph wiring 시 결정 — 별도 필드 도입은 미리 하지 않음).
- `chat/services/agent/tools.py` — `Tool` dataclass + 모듈-수준 registry.
  - `Tool.input_schema: Optional[Mapping[str, FieldSpec]]` — `None` 은 "스키마 검증 생략, callable 이 자체 검증" 의 의도된 escape hatch. `run_workflow` 처럼 입력 형태가 외부 도메인에 따라 달라지는 도구가 이 모드를 쓴다.
  - 등록되는 첫 도구 세 개:
    - `retrieve_documents` — schema `{'query': FieldSpec(type='text', required=True)}`. `chat.services.single_shot.retrieval.retrieve_documents`.

    - `run_workflow` — `input_schema=None` (raw). callable 이 `arguments['workflow_key']` + `arguments['input']` 을 추출해 `chat.workflows.domains.dispatch.run` 으로 위임. dispatch 가 자체 `UNSUPPORTED / MISSING_INPUT / INVALID_INPUT` 으로 잘못된 입력을 잡아 Observation 으로 흡수.
- `chat/services/agent/react.py` — ReAct loop (think → choose tool → execute → observe → decide). 한 step 한 LLM 호출 + max_iterations 안전판.

- `chat/services/agent/prompts.py` — system prompt 빌더 + JSON 응답 스펙. 프롬프트 본문은 외부 파일(`assets/prompts/chat/agent_react.md`) + Phase 1 `prompt_registry` 등록.
- 단위 테스트 (mocked LLM + mocked tool callable):
  - tool registry 등록 / 알 수 없는 이름 거부 / 입력 스키마 검증
  - ReAct happy path (1 tool 호출 후 final_answer)
  - multi-hop (도구 두 번 호출 후 종료)
  - max_iterations 초과 → `UPSTREAM_ERROR("max_iterations_exceeded")` 또는 `NOT_FOUND("insufficient_evidence")` (정책은 §6 에서 결정)
  - tool 예외 발생 → 다음 iteration 진행 / 연속 실패 시 종료
  - LLM JSON 파싱 실패 → UPSTREAM_ERROR

## 핵심 설계 결정

### 1. agent / workflow / single_shot 의 책임 분리 (설계 §5)

```
single_shot   — 자료 한 번 찾고 LLM 한 번 — 정답이 거의 그대로 있는 질문
workflow      — 입력이 정해진 절차형 계산 — 출력이 결정적
agent         — 어느 자료/도구를 어떤 순서로 쓸지 유동적 — 도구를 반복 호출
unsupported   — 근거 없음 / 입력 없음 / 추정 금지
```

이 위계는 Phase 7-2 의 라우터 정책으로 강제된다 (현재 코드 키워드 fallback 의 `agent_keyword` 가 거기에 닿음). 7-1 에서는 코드 코멘트로 원칙을 명문화하고, runtime 자체가 이 원칙을 위반하지 않도록 도구 카탈로그를 좁힌다.

### 2. AgentState (설계 §9)

```python
@dataclass
class AgentState:
    question: str                            # 사용자 원본 질문 (raw — Phase 7-1)
    history: list[dict]                      # 최근 history (원문 그대로 — agent 가 직접 보지는 않음)
    agent_goal: str = ''                     # 첫 think 단계가 채움 (자유 문자열)
    observations: list[Observation] = field(default_factory=list)
    tool_calls: list[ToolCall] = field(default_factory=list)
    iteration_count: int = 0
    final_answer: Optional[str] = None
    termination: Optional[AgentTermination] = None
    error: Optional[str] = None              # 치명적 오류 메시지
```

**`question` 필드의 의미**: Phase 7-1 에선 **사용자 원본 입력 그대로**. history-aware rewrite 의 적용 위치는 Phase 7-2 의 결정 — 두 가지 옵션:

- (a) graph node 가 dispatch 직전 `rewrite_query_with_history` 를 돌려 결과를 `state['question']` 자리에 그대로 주입 (table_lookup 의 `workflow_node` 와 동일 패턴). agent 입장에선 여전히 단일 필드.
- (b) `AgentState` 에 `search_query` 같은 별도 필드를 추가해 raw 와 rewrite 를 분리.

**Phase 7-1 은 (a) 기준으로 설계** — 즉 별도 필드는 미리 도입하지 않는다 (YAGNI). 7-2 에서 (a) 로 충분치 않다는 게 드러나면 그때 (b) 로 확장. 이 결정은 dev log 에도 명시.

`Observation` / `ToolCall` 은 **요약 형태** 로 저장한다 — tool 원문 응답을 통째로 쌓으면 다음 iteration 의 LLM 컨텍스트가 폭주한다.

```python
@dataclass(frozen=True)
class ToolCall:
    name: str
    arguments: Mapping[str, Any]

@dataclass(frozen=True)
class Observation:
    tool: str
    summary: str          # 100~400 자 수준의 핵심 요약
    is_failure: bool      # 도구 호출 자체가 실패한 경우 True
```

요약 정책: tool 결과가 list/dict 면 길이만 / 첫 N 항목 / 핵심 키만 추출. 자연스러운 한국어로 한 줄~세 줄.

### 3. Tool 인터페이스 (설계 §8)

```python
@dataclass(frozen=True)
class Tool:
    name: str
    description: str
    input_schema: Optional[Mapping[str, FieldSpec]]   # None = schema-free (raw mode)
    callable: Callable[[Mapping[str, Any]], Any]
    summarize: Callable[[Any], str]                    # tool 결과 → Observation.summary 변환
```

**`input_schema` 두 모드**:

- **schema 모드** (`Mapping[str, FieldSpec]`) — Phase 6-2 FieldSpec 어휘를 그대로 재사용. registry 의 `call(name, arguments)` 가 호출 직전 schema 검증을 수행, 실패 시 callable 을 부르지 않고 `Observation(is_failure=True, summary='input invalid: ...')` 만 남기고 다음 iteration.
- **raw 모드** (`None`) — "스키마로 강제할 수 없는 도구" 에만 의도적으로 적용하는 escape hatch. registry 는 generic 검증을 스킵하고 `arguments` 를 그대로 callable 에 넘긴다. callable / 그 아래 도메인이 자체 status (`UNSUPPORTED / MISSING_INPUT / INVALID_INPUT`) 로 잘못된 입력을 걸러야 하며, 그 결과는 Observation 으로 흡수된다.

`summarize` 는 도구별로 정의 (긴 문서 리스트 → "5건, 첫 출처: ...").

등록은 모듈-수준 register. 알려진 도구 이름 외에는 LLM 이 호출해도 거부 → "그 이름 없음" observation 만 남기고 다음 iteration 진행.

**초기 3 도구**:

| 이름 | input_schema | 호출 | 요약 |
|---|---|---|---|
| `retrieve_documents` | `{'query': FieldSpec(type='text', required=True)}` | `chat.services.single_shot.retrieval.retrieve_documents(query)` | "<N>개 청크, 첫 출처: <파일명>" |
| `find_canonical_qa` | `{'query': FieldSpec(type='text', required=True)}` | `chat.services.single_shot.qa_cache.find_canonical_qa(query)` | "<N>개 후보, top similarity=<x>" |
| `run_workflow` | `None` (raw) | `chat.workflows.domains.dispatch.run(arguments['workflow_key'], arguments.get('input', {}))` | `WorkflowResult.status` + value 짧게 |

`run_workflow` 가 raw 인 이유: 두 번째 인자(`input`) 의 형태가 `workflow_key` 에 따라 완전히 달라진다 (date_calculation 은 `{start, end, unit}`, amount_calculation 은 `{values, op}`, table_lookup 은 `{query}`). 한 번에 한 schema 로 강제할 수 없으니 callable 안에서 `dispatch.run` 으로 위임하고, 잘못된 입력은 Phase 6 의 status 로 흘러나오게 한다.

테스트로 고정할 것: schema 모드 도구가 잘못된 입력을 거부 → callable 이 호출되지 않음 / raw 모드 도구는 우회 검증 없이 callable 까지 도달 / 알려지지 않은 도구 이름은 양쪽 다 거부.

### 4. ReAct loop (설계 §10)

한 iteration 의 형태:

```
LLM (system + state summary + tool catalog) →
  JSON: {"thought": "...", "action": "tool_name" | "final_answer", "arguments"?: {...}, "answer"?: "..."}
↓
action == "final_answer"  → AgentTermination.final_answer 로 종료, to_workflow_result(...) → WorkflowResult.ok(value=answer, details=...)
action ∈ tools             → registry.call(tool_name, arguments) → Observation 추가
action 외                  → Observation(is_failure=True, summary='unknown action')
↓
iteration_count += 1
iteration_count >= max_iterations → AgentTermination.max_iterations_exceeded → to_workflow_result(...) → UPSTREAM_ERROR
연속 실패 (≥3)             → AgentTermination.no_more_useful_tools → to_workflow_result(...) → NOT_FOUND
```

**LLM JSON 응답 강제**: system prompt 에 "JSON 한 줄, 그 외 토큰 금지" + `_parse_json_object` 가 fence/공백 관용적으로 처리. 파싱 실패는 한 번 retry 후에도 안 되면 `UPSTREAM_ERROR` 종료.

**프롬프트 본문**은 `assets/prompts/chat/agent_react.md` 외부화 + Phase 1 registry 에 `chat-agent-react` 로 등록. 운영자가 도구 리스트 / 종료 정책 문구를 BO 에서 튜닝 가능.

### 5. 결과 상태 매핑 (설계 §12)

내부 종료 사유 → 사용자 facing status:

| internal `AgentTermination` | 사용자 status | reason 예시 |
|---|---|---|
| `final_answer` | `OK` | (없음) — `value` 가 답 |
| `max_iterations_exceeded` | `UPSTREAM_ERROR` | "도구를 너무 많이 사용했어요. 잠시 후 다시 시도해 주세요." |
| `no_more_useful_tools` | `NOT_FOUND` | "더 알아볼 도구가 남지 않아 충분한 답을 만들 수 없었습니다." |
| `insufficient_evidence` | `NOT_FOUND` | "근거가 부족해 답을 만들기 어려웠습니다. 자료가 충분한지 확인해 주세요." |
| `fatal_error` | `UPSTREAM_ERROR` | "일시적인 오류가 발생했습니다. 잠시 후 다시 시도해 주세요." |

`UNSUPPORTED` 는 agent 자체에서 직접 만들지 않는다 — 라우팅 단계에서 결정되는 게 정책상 맞다 (설계 §5-3, §6-2). agent 가 시작했는데 unsupported 로 끝나는 건 상위 라우팅의 실패라고 본다 → 이 PR 의 회귀 테스트로 가드.

### 6. 결과 타입 — `WorkflowResult` 재사용 확정

설계 §12 는 status 어휘만 공유하고 별도 타입을 권장하지만, **Phase 7-1 은 `WorkflowResult` 를 그대로 반환형으로 사용**하기로 확정. `chat/services/agent/result.py` 에는 새 공개 dataclass 를 만들지 않고:

- `AgentTermination` enum — 내부 종료 사유.
- `to_workflow_result(termination, *, value=None, reason='')` 헬퍼 — termination → user-facing `WorkflowResult` 변환.

이 선택의 trade-off:

- **장점** — Phase 6-3 의 reply 분기를 재사용. `details={'tool_calls_count': N, 'sources_used': [...], 'termination': '...'}` 같은 메타는 details dict 에 자유롭게 실어 보내고, reply 가 status 만 보고 기본 카피를 결정.
- **어색함** — 이름이 `WorkflowResult` 인데 agent 가 반환한다. dev log §"의도된 어색함" 에 기록.
- **out-of-scope (이후 리팩터 후보)** — `BaseResult` 추출 + `WorkflowResult` / `AgentResult` 가 이를 상속하는 구조. 7-1 에서는 시기상조.

Scope 에 적힌 "AgentResult" 표현이 본 결정과 어긋나지 않도록 Scope 본문도 동일 문구로 정렬됨 (위 §Scope 참고).

### 7. 호출 비용 통제 (설계 §11)

- `MAX_ITERATIONS = 4` (코드 상수, 추후 BO 토글 후보).
- `MAX_CONSECUTIVE_TOOL_FAILURES = 3` — 연속 도구 실패 시 즉시 종료.
- `LLM_RETRY_ON_PARSE_ERROR = 1` — JSON 파싱 한 번만 retry.
- 각 LLM 호출마다 `record_token_usage` (Phase 4-3 / 6-2 / 6-3 와 같은 패턴).
- agent 한 턴 최대 LLM 호출 = `MAX_ITERATIONS + 1` (마지막 final_answer 변환 포함) ≈ 5 회. 토큰 가시성을 위해 호출마다 `purpose='agent_step'` 같은 메타 필드를 추가하는 건 out-of-scope (TokenUsage 모델 변경이라 7-2+).

## 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| LLM 이 catalog 외 도구 이름을 호출 | `tools.call(name, …)` 가 미등록 시 `is_failure=True` Observation 만 남기고 다음 iteration 진행. 연속 실패 시 종료 |
| LLM 이 무한 루프 (같은 tool · 같은 입력 반복) | `tool_calls` 에 `(name, arguments)` 해시 누적, 동일 호출 3 회 → `no_more_useful_tools` 종료 |
| Observation 폭주로 컨텍스트 초과 | `Observation.summary` 길이 상한 (예: 600 자), 한 번 truncate 후 LLM 컨텍스트 빌드 시 최근 N 개만 노출 |
| 도구 호출 결과가 너무 길어서 summarize 가 의미 잃음 | tool 별 summarize 함수가 도메인 특화 — list 면 길이만, dict 면 핵심 키만 |
| LLM JSON 응답이 자꾸 깨짐 | system prompt 에 "JSON 한 줄, 그 외 토큰 금지" 강조 + `_parse_json_object` 가 fence/whitespace 관용 + 1회 retry 후에도 실패 시 `UPSTREAM_ERROR` |
| Phase 7-1 코드가 import 만 돼도 다른 모듈에 영향 | `chat/services/agent/__init__.py` 가 다른 모듈을 import 하지 않게 하고(부작용 차단), graph / view 어떤 곳에서도 agent 패키지 reference 없음을 PR 검수에서 확인 |
| 결과 타입을 `WorkflowResult` 로 재사용한 게 의미적으로 어색 | dev log 에 "의도된 어색함" 으로 명시 + Phase 7-2 의 reply 가 status 만 보면 되도록 분기 단순. 후속 `BaseResult` 추출은 별도 리팩터 |

- `chat/services/agent/` 패키지가 import 가능하고 모듈 6 개(`__init__ / state / tools / tools_builtin / prompts / react / result`) 존재.

- AgentState / Tool / ReAct loop / Termination 매핑 단위 테스트 ~30 건 green.
- 도구 세 개(`retrieve_documents`, `find_canonical_qa`, `run_workflow`) 가 registry 에 등록되고 mock 으로 호출 가능. `run_workflow` 가 잘못된 key 를 받으면 dispatch 의 status 가 Observation 으로 그대로 surface.

- agent runtime 이 LLM 호출 시 `record_token_usage` 를 호출.
- chat UI / graph / BO / migration 변경 없음 (회귀 0).
- README + Phase 7-1 dev log 반영.
- Phase 7-2 의 진입점(`run_agent(question, history)` → `WorkflowResult`) 시그니처가 안정화 — 7-2 가 `agent_node` 로 wrapping 가능.
