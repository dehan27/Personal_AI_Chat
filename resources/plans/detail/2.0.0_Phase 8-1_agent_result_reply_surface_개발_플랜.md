# 2.0.0 Phase 8-1 — Agent Result / Reply Surface 정리 개발 플랜

## Context

Phase 7-4 머지 직후 상태:

- `chat/services/agent/result.py` 의 `to_workflow_result(...)` 가 `AgentTermination` 을 **`WorkflowResult` 그대로** 변환해 반환. agent 전용 결과 dataclass 없음.
- `chat/services/agent/reply.py` 의 `build_reply_from_agent_result(result: WorkflowResult)` 가 status 만 보고 한국어 reply 문자열 생성.
- `chat/graph/nodes/agent.py` 가 `QueryResult(reply=..., sources=[], total_tokens=0, chat_log_id=None)` — **sources 빈 리스트 / chat_log_id None** 으로 고정.
- 결과적으로 사용자가 답변을 봐도 (a) 어떤 문서가 근거였는지 / (b) agent 가 어떤 도구를 썼는지 알 수 없음. workflow / single_shot 답변과의 형식 차이도 큼 (single_shot 은 `sources` 정상 노출).
- 7-1 시점부터 "WorkflowResult 재사용" 은 의도된 어색함으로 두고 갔지만, 7-2 의 reply 포맷터 + 7-4 의 failure_kind 까지 쌓이면서 agent-only 의미가 늘어남 — 더 이상 workflow 와 같은 그릇에 담는 게 깔끔하지 않음.

Phase 8-1 의 목표는 **agent 결과 구조와 사용자-facing 표면을 정리** 해 8-2 (TokenUsage purpose) / 8-3 (BO 운영 제어) 의 기반을 만드는 것. 단 **대규모 리팩터는 피하고** workflow/single_shot 의 public contract 는 깨지 않는다 (Phase 8 §8-1 회귀 0 우선).

> 이 plan 은 Phase 8 설계 §5 (Agent Result / Reply Surface 정리) 에 직접 매핑. §5-2 의 6 가지 후보 작업과 §5-3 의 4 가지 설계 질문이 그대로 본 plan 의 §핵심 설계 결정으로 들어옴.

## 핵심 설계 결정

설계 §5-3 의 4 가지 질문 + §5-4 권장 방향에 대한 본 plan 의 답:

### 1. `AgentResult` 신규 vs `WorkflowResult` 재사용 — **AgentResult 신규 도입**

설계 §5-4 는 "BaseResult 는 도입하되 대규모 리팩터는 피한다" 권장. 본 plan 은 그 방향을 따라:

- **`BaseResult`** — 공통 필드 (`status / value / details`) 만 명시한 Protocol/abstract. 코드 30 줄 이내.
- **`WorkflowResult`** — `BaseResult` implement. **Public 시그니처 / 팩토리 / 필드 / __dataclass_fields__ 변경 0**. 외부에서 보면 7-4 와 동일.
- **`AgentResult`** — `BaseResult` implement + 추가 필드 (tool_calls / sources / termination). reply layer + agent_node 에서만 점진적으로 사용.

이 분할의 trade-off:
- ✓ workflow 호출부 (~50곳 단위 테스트 포함) 한 줄도 안 바뀜.
- ✓ agent 가 자신의 의미 (tool_calls, sources) 를 1급 필드로 표현.
- ✓ 8-2 / 8-3 / 후속 Phase 가 AgentResult 에 필드 추가하기 쉬움.
- ✗ 두 result 타입 공존 — 다만 reply layer 는 단일 타입 (`AgentResult`) 만 받음. agent reply 와 workflow reply 가 이미 분리된 모듈 (`chat/services/agent/reply.py` vs `chat/workflows/domains/reply.py`) 이라 한 함수가 둘 다 받을 일 없음. `BaseResult` 는 status 어휘 공유 / 후속 확장 용도이지 reply 진입점 통합 목적이 아님.

### 1-1. trace 생성 계약 — `Observation` 단일 소스 (P2-1 보강)

**문제**: 현재 `AgentState` 는 `tool_calls` 와 `observations` 를 별도 리스트로 들고 있는데, 이 둘은 **1:1 정렬이 보장되지 않는다**. 구체적으로:

| step 종류 | `record_tool_call` | `add_observation` | 결과 |
|---|---|---|---|
| 정상 도구 호출 | ✓ | ✓ | 1:1 매칭 |
| `invalid_args` (non-Mapping) | ✗ | ✓ | obs 만 |
| `repeated_call` 차단 | ✗ | ✓ | obs 만 |
| JSON parse 실패 retry | ✗ | ✓ (`tool='_llm'`) | obs 만 |

`tool_calls` 와 `observations` 의 길이도 다르고 인덱스도 안 맞는다. `ToolCallTrace = ToolCall + Observation 페어` 로 재구성하면 invalid_args / repeated_call 의 시도 인자가 사라진다 — 운영자가 "LLM 이 어떤 args 로 잘못 시도했는지" 못 봄.

**해법**: `Observation` 에 `arguments: Mapping[str, Any]` 필드 추가. 모든 add_observation 호출부 (`tools.call` / `react.py` 의 invalid_args / repeated_call / _llm parse 실패) 가 시도된 args 를 obs 에 함께 박는다. trace 는 `state.observations` 만 walk 해 `ToolCallTrace` 리스트로 변환.

장점:
- step 단위 1:1 매칭 보장 (관심 단위 = step = observation).
- invalid_args / repeated_call 의 시도 args 보존 — 8-3 의 BO 운영 제어에서 "최근 실패 케이스" 표시 시 필수.
- `state.tool_calls` 는 그대로 유지 (`repeated_call_count` 가드 등 기존 사용처 보존), trace 생성에는 사용 안 함.

`Observation.arguments` 의 의미: **호출 시점의 raw args** (정규화 전). non-Mapping 으로 시도된 invalid_args 케이스는 `arguments={}` 로 두고 summary 에 원형 정보가 들어가는 방식 — 시도된 형식 정보는 `failure_kind='invalid_args'` 와 summary 텍스트로 전달.

`tool='_llm'` (parse 실패) 도 trace 에 포함 — 운영자가 디버깅 시 LLM 이 JSON 못 만든 step 도 봐야 유용. arguments 는 `{}`.

### 2. sources 형식 — **`single_shot` 의 `{'name', 'url'}` dict 와 호환**

설계 §5-3 의 두 번째 질문 ("기존 ChatLog.sources 와 같은 형식?") 답: **예**.

- `AgentResult.sources: Tuple[SourceRef, ...]` — typed dataclass.
- `SourceRef.to_dict()` → `{'name': ..., 'url': ...}` — `QueryResult.sources` 에 그대로 박을 수 있음.
- `AgentResult.sources_as_dicts() -> List[Dict[str, str]]` 메서드 — `[s.to_dict() for s in self.sources]` 를 한 줄로 캡슐화. `agent_node` 등 호출부는 list comprehension 안 쓰고 helper 만 사용 → 변환 로직 한 곳.

### 3. sources 노출 정책 — **단일 정책 (P2-2 / P2-3 보강)**

설계 §5-3 의 세 번째 질문 ("agent 가 본 모든 chunk vs 실제 사용한 것?") 답.

**확정된 단일 정책**:

1. **수집 위치**: `_retrieve_callable` 이 결과 dict 의 `'evidence'` 키에 **`hits[0]` 한 건만** SourceRef 로 넣는다. `hits[1:]` 는 sources 후보가 아님 — top-1 만 retrieval 의 "이 step 이 본 대표 출처".
2. **부착 위치**: `tools.call` 이 result dict 의 `'evidence'` 키를 보고 `Observation.evidence` 로 자동 박음. retrieve 외 도구는 evidence 미세팅 → 빈 튜플.
3. **AgentResult.sources 수집**: `to_agent_result(state)` 가 `state.observations` 를 walk 하면서 evidence 가 비어있지 않은 obs 의 SourceRef 를 dedup (`(name, url)` 튜플 키) 해 `Tuple[SourceRef, ...]` 로 모음.
4. **low_relevance 제외**: `obs.is_failure and obs.failure_kind == 'low_relevance'` 인 obs 의 evidence 는 수집 시 **건너뜀** — 관련 없는 자료를 출처로 노출하지 않기 위해.
5. **status 무관 노출**: `agent_node` 가 `result.sources` 를 status 분기 없이 그대로 `QueryResult.sources` 에 박는다 (P2-3 보강 — 정책 충돌 해소). 단:
   - OK: 의미 매치 retrieve 의 evidence 가 정상적으로 모임 → 사용자에게 출처 패널 표시.
   - NOT_FOUND (low_relevance 누적): 모든 retrieve 가 low_relevance 라 evidence 자동 제외 → 자연스럽게 빈 리스트.
   - NOT_FOUND (max_iter / consecutive_failures): 일부 retrieve 가 success 였으면 그 출처는 노출. "자료는 봤지만 정리 못 함" 의 정직한 표현.

이 단일 정책의 결과:

| 시나리오 | retrieve 발생 | 모든 retrieve가 low_relevance? | sources 결과 |
|---|---|---|---|
| 우주여행 비교 (NOT_FOUND) | 3~5회 | 예 | `[]` (자동) |
| 결혼 경조금 비교 OK | 2~3회 | 아니오 | top-1 dedup 출처 1~2개 |
| 복리후생 규정 비교 NOT_FOUND (max_iter) | 6회 | 아니오 (모두 success) | dedup 출처 (보통 같은 문서 1개) |
| invalid_args / repeated_call 만 | retrieve 없음 | — | `[]` |

이유 (대안 미채택):
- "실제 final_answer 가 인용한 도큐먼트만" 추출하려면 final_answer 텍스트와 chunk 매핑 — 별도 LLM 호출 또는 휴리스틱이라 8-1 범위 초과.
- "모든 hits" 노출은 출처 리스트 폭주 (top-N=5 × retrieve N회 = 최대 30 개).
- top-1 만 한 곳 (`_retrieve_callable`) 에서 결정해 `to_agent_result` 는 단순 dedup + 필터 — 정책 분산 X.

### 4. tool trace 사용자 노출 vs BO/log 전용 — **데이터는 결과에 담되 사용자-facing UI 는 미노출**

설계 §5-3 의 네 번째 질문 + §5-4 권장 ("길게 노출하지 않고 요약 또는 BO/로그용") 답:

- `AgentResult.tool_calls` — 1급 필드로 데이터는 보존.
- `agent_node` 는 tool_calls 를 `QueryResult` 에 박지 **않음** — 사용자에게 노출 0 (현재 UI 가 도구 목록 표시 안 함).
- BO / 로그 / 8-3 의 운영 제어 페이지에서 활용할 데이터로 보존.
- 즉 8-1 은 "데이터 구조 마련" 까지, "노출 UI" 는 8-3 또는 이후.

### 5. `to_workflow_result` 어댑터 유지로 호환성 보장

`run_agent` 의 반환을 `AgentResult` 로 바꾸면 일부 단위 테스트 (`test_agent_react.py` 등) 가 `WorkflowResult` 를 가정하는 부분이 영향. 두 가지 길:

- (a) 모든 호출부 일제히 `AgentResult` 로 전환 — 영향 범위가 좁아 (graph node 1곳, reply 1곳, 단위 테스트 몇 곳) 가능.
- (b) `AgentResult.to_workflow_result()` 어댑터 메서드를 두고 호출부 점진 전환.

## 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| `BaseResult` 도입이 `WorkflowResult` 동작을 미세하게 바꿔 ~50개 단위 테스트 중 일부 깨짐 | Step 1 의 회귀 가드 (3 케이스) + 전체 chat.tests 통과로 1차 봉쇄. abstract 가 아니라 Protocol 로 가면 implement 표시도 명시적이지 않아 안전. structural typing 으로만 |
| `AgentResult` 와 `WorkflowResult` 두 타입 공존 → reply layer 가 둘 다 처리하느라 분기 늘어남 | agent reply (`chat/services/agent/reply.py`) 와 workflow reply (`chat/workflows/domains/reply.py`) 는 이미 분리된 모듈 — 한 함수가 둘 다 받지 않음. `build_reply_from_agent_result` 는 `AgentResult` 만, `build_reply_from_result` 는 `WorkflowResult` 만. 두 타입 공존이 reply 에 새 분기 추가 0 |
| `Observation.evidence` 필드 추가가 `state.add_observation` 호출부 (`react.py`) 에서 evidence 안 넘기는 케이스 시 빈 튜플 default — 직관적이지만 tests 의 직접 `Observation(...)` 인스턴스화 영향 가능 | dataclass field default `()` + kw-only 로 두어 positional 깨지지 않게 |
| sources dedup 정책이 fragile — 같은 도큐먼트가 여러 retrieve 에 걸쳐 있으면 dedup 키 (`document_id` or `name`) 결정 필요 | `SourceRef.to_dict()` 의 키는 `(name, url)` tuple 로 dedup. document_id 가 없는 형태 (예: 외부 link) 도 안전 |
| agent 답변에서 sources 노출이 늘어나 UI 가 "관련 없는 자료" 도 출처로 표시할 수 있음 (특히 NOT_FOUND 시) | low_relevance failure 가 마킹된 obs 의 evidence 는 sources 수집에서 제외 — `to_agent_result` 가 `obs.is_failure and obs.failure_kind == 'low_relevance'` 인 경우 evidence 무시 |
| 본 plan 머지 후 8-2 가 시작될 때 `AgentResult.tool_calls` 형태 변경이 필요하면 8-2 에서 또 바꿔야 함 | 8-1 단계에서 8-2 의 token_usage purpose 매핑 흐름을 미리 검토 — `ToolCallTrace.failure_kind` 와 `purpose` 가 호환되도록 (예: `agent_step` purpose 가 모든 retrieve obs 에 매핑) |
| `__init__.py` docstring 변경만으로 `tools_builtin` 의 import 부작용이 영향받는 케이스 | docstring 만 변경, 코드 변경 0 — 영향 없음 |
