# 2.0.0 Phase 6-1 — Workflow Dispatch 인프라 + date_calculation 개발 플랜

## Context

Phase 5 에서 `chat/workflows/core/` 를 깔아둔 뒤, `chat/workflows/domains/` 은 아직 비어 있다. Phase 4 라우팅은 질문을 `single_shot / workflow / agent` 로 분류하지만 **`workflow` 는 여전히 single_shot 으로 포워딩**되고 있다. Phase 5 core 는 실제 사용처가 없는 상태.

Phase 6 설계(`resources/plans/2.0.0_Phase 6_범용_Workflow_Domain_개발_설계.md`)는 회사 전용 계산(퇴직금·연차)이 아니라 **질문 유형(`date_calculation`, `amount_calculation`, `table_lookup`, …)** 단위의 generic workflow 도메인을 올리는 것을 목표로 한다. 설계는 Step 0~5 까지의 흐름을 하나로 묶어 제시하지만, 실행 관점에서 규모가 커서 세 개로 쪼갠다:

- **Phase 6-1 (이 플랜)** — 표준 정리 + dispatch 인프라 + 첫 generic workflow(`date_calculation`). 인프라가 실제로 동작한다는 걸 end-to-end 로 증명.
- **Phase 6-2** — `amount_calculation` 추가.
- **Phase 6-3** — `table_lookup` 추가 + 남은 정책 다듬기.

Phase 6-1 의 핵심 약속: **등록된 workflow 가 없는 질문은 기존처럼 `single_shot` 으로 포워딩** 된다. 즉 operator 가 BO 에서 workflow_key 를 찍어주지 않는 한 회귀 0.

## 핵심 설계 결정

### 1. workflow_key 는 문자열 상수 — enum 으로 강제하지 않는다

```python
# chat/workflows/domains/registry.py
_REGISTRY: dict[str, WorkflowEntry] = {}

@dataclass(frozen=True)
class WorkflowEntry:
    key: str
    title: str             # BO 드롭다운 표시
    description: str
    status: str            # 'stable' | 'beta' — BO 에서 표시용
    factory: Callable[[], BaseWorkflow]   # 호출 시 새 인스턴스
```

- 이유: Phase 6-2, 6-3 에서 key 가 늘어나고, 운영자가 BO 에서 새 key 를 입력할 수 있어야 유연.
- 검증: `dispatch.run(key, raw)` 가 registry 에 없는 key 를 받으면 `WorkflowResult.unsupported(reason='unknown workflow key: ...')`.

### 2. dispatch 는 얇게, workflow 는 pure

```python
# chat/workflows/domains/dispatch.py
def run(workflow_key: str, raw: Mapping[str, Any]) -> WorkflowResult:
    entry = registry.get(workflow_key)
    if entry is None:
        return WorkflowResult.unsupported(f'등록되지 않은 workflow: {workflow_key!r}')
    workflow = entry.factory()
    return core.run_workflow(workflow, raw)
```

- `run_workflow` 는 Phase 5 `base.py` 의 레퍼런스 러너. `prepare → validate → execute` 4 단계 계약.
- dispatch 는 key 해석 + 인스턴스 생성만. 도메인 로직 금지.

### 3. graph workflow_node 동작

```python
def workflow_node(state: GraphState) -> GraphState:
    key = state.workflow_key or ''
    if key and registry.has(key):
        result = dispatch.run(key, state.workflow_input or {})
        return _project_result_into_state(state, result)
    # 폴백: workflow_key 가 없거나 미등록이면 기존 single_shot 경로.
    return single_shot_node(state)
```

- `state.workflow_input` 은 Phase 6-1 에서는 `{}` 로 시작 — raw 입력을 자연스럽게 추출하는 로직은 6-2 이후. `date_calculation` 만 수동 테스트용으로 쓰인다.
- 실사용 시: 운영자가 BO RouterRule 에서 `pattern="며칠"` + `workflow_key="date_calculation"` 로 등록하면 질문 자체에서 날짜가 뽑히지 않으니, 단순히 **"입력 부족" 안내** 로 answer 조립. 즉 **실질적 사용자 답변 흐름은 Phase 6-2 에서 input extraction 설계 시 완성**된다.
- 그럼에도 Phase 6-1 가 가치있는 이유: 인프라·route_reason·workflow_key log / BO 에서 지정 가능 / dispatch 경로 검증 가능.

### 4. RouteDecision 과 GraphState 확장

```python
@dataclass(frozen=True)
class RouteDecision:
    route: str
    reason: str
    matched_rules: List[str] = field(default_factory=list)
    workflow_key: str = ''          # Phase 6-1 추가
```

- 기본값 빈 문자열 → 기존 호출부 호환.
- `route != 'workflow'` 일 때는 항상 `workflow_key=''`.

GraphState 도 동일. 추가 필드 있음으로 인한 회귀는 없음 (serializer 가 없는 내부 dataclass).

### 5. RouterRule 마이그레이션

- 새 컬럼: `workflow_key = CharField(max_length=64, blank=True, default='')`.
- `blank=True` 라 기존 규칙은 빈 값으로 저장.
- `chat/services/question_router._match_db_rules` 반환에 `workflow_key=rule.workflow_key` 포함.
- BO 폼의 `route` 드롭다운 아래에 `workflow_key` 드롭다운 (registry 기반 옵션 목록).

### 6. 표준 정리 — 키워드는 유지, 예시만 정리

설계 §10-1 은 `퇴직금 / 연차 계산` 같은 회사 전용 키워드를 정리하라고 하지만, 이 키워드들은 이미 Phase 4-1 에서 fallback 역할을 하는 **코드 상수**(`WORKFLOW_KEYWORDS`, `AGENT_KEYWORDS`) 다. 지우면 실사용 질문 분류가 무너짐.

Phase 6-1 의 정리 범위는:
- 상수 **값은 그대로 두고** 모듈 docstring 에 "특정 회사 규정과 무관한 패턴 신호일 뿐, 회사 도메인 workflow 가 아니라 단지 single_shot fallback 구별용" 임을 명확히.
- `README.md §3-1` question_router 설명을 "workflow route 를 맞추는 키워드 신호" → "Phase 6 이후 workflow_key 를 결정하는 건 BO RouterRule" 로 보강.
- Phase 6-2 부터 실제로 `parse_date` 입력 추출을 넣을 때, 이 키워드들이 필요 없어진다면 그때 정리.

### 7. date_calculation 동작 스펙

입력 (`Mapping[str, Any]`):
- `start: str` — 시작 날짜 문자열(core `parse_date` 지원 포맷)
- `end: str` — 종료 날짜 문자열
- `unit: str` — `'days' | 'months' | 'years'` (기본 `'days'`)

실행:
1. `require_fields({'start', 'end'})` → missing 이면 `missing_input`.
2. `parse_date(start)` / `parse_date(end)` 실패 → `invalid_input(errors=[...])`.
3. `ensure_date_order(start, end)` 실패 → `invalid_input`.
4. `unit` 에 맞춰 `days_between` / `months_between` / `years_between` 호출.
5. `WorkflowResult.ok(value=<int>, details={'start': ..., 'end': ..., 'unit': ...})`.

응답 포맷 (`chat/graph/nodes/workflow.py` 에서 조립):
- OK: `"{start} 부터 {end} 까지 {N}{unit_한글} 입니다."`
- MISSING_INPUT: `"계산하려면 {missing_field_list} 정보가 필요합니다."`
- INVALID_INPUT: `"{errors}"` 조합.
- UNSUPPORTED: `"현재 지원하지 않는 작업입니다."`

LLM 호출 없음. Phase 6-1 범위에선 workflow 는 **결정적 계산기**.

| 리스크 | 대응 |

| workflow_key 가 빈 문자열인 RouterRule 이 섞여서 dispatch 호출 자체가 불필요하게 돎 | workflow_node 가 `key` 빈 문자열이면 즉시 single_shot 폴백 (dispatch 호출 전 가드) |

| Phase 6-2 이후에 workflow_key enum 화 유혹 | 문자열 유지 원칙 README 에 명시. BO 에서 새 key 입력 가능한 구조를 잃지 않도록 |
