# 2.0.0 Phase 7-4 — Agent Decision Policy & Retrieval Relevance Awareness 개발 플랜

## Context

Phase 7-3 머지 직후 smoke 검증에서 **windowing 본 무대 밖** 의 결함 두 가지가 드러남. 7-3 의 retrieve summary 변경과 무관한 별도 정책 결함이라 7-3 머지에는 영향 없었지만, 운영 사용성에는 영향. 본 PR 이 그 둘을 정리.

### Defect 1 — 관련 없는 query 에서 무한 retrieve 회로

질문: `우주여행 비교` (회사 문서에 우주여행 관련 자료 없음).

```
step 0: retrieve_documents '우주여행 비교'         → 5건  (is_failure=False)
step 1: retrieve_documents '우주여행 비용 비교'     → 5건
step 2: retrieve_documents '우주여행 프로그램 비교' → 5건
step 3: retrieve_documents '우주여행 비용 및 프로그램 비교' → 5건
step 4: retrieve_documents '우주여행 프로그램 및 비용' → 1건
step 5: retrieve_documents '우주여행 비교 및 비용'   → 5건  → MAX_ITERATIONS_EXCEEDED
→ status=upstream_error, "도구를 너무 많이 사용했어요. 잠시 후 다시 시도해 주세요."
```

원인: retriever 가 cosine similarity 상 가장 가까운 청크 N건을 **항상 비실패로 반환** (실제 관련성 없어도). 7-3 의 windowing 은 정상 동작 — 청크에 `우주여행` 토큰이 없으니 미매치 fallback 으로 청크 첫 N자 (관련 없는 헤더) 노출. 그런데 LLM 은 summary 의 "5건 검색됨" 만 보고 "결과 있다, query 만 다듬자" 로 해석하고 무한 retrieve. 안전판인 `MAX_ITERATIONS=6` 에 걸려 사용자에게 일시적 오류 카피가 나가 운영적으로 어색.

기대: 한두 번의 retrieve 후 "스니펫에 query 키워드가 하나도 안 보이네 → 자료 못 찾았다" final_answer 로 정직하게 종료 (status=OK + "자료를 찾지 못했습니다" 류).

### Defect 2 — 같은 (tool, args) 핑퐁

질문: `복리후생 규정 비교` (broad query — 포커스 부족).

```
step 0: retrieve_documents '복리후생 규정 비교'      → 3건
step 1: retrieve_documents '복리후생 규정 세부사항'  → 4건
step 2: retrieve_documents '복리후생 규정 비교'      → 3건  ← step 0 과 동일
step 3: retrieve_documents '복리후생 규정 세부사항'  → 4건  ← step 1 과 동일
step 4: retrieve_documents '복리후생 규정 비교'      → 3건  → MAX_REPEATED_CALL=3 trigger
→ status=not_found, "더 알아볼 도구가 남지 않아 충분한 답을 만들 수 없었습니다."
```

원인: LLM 이 두 query 사이를 핑퐁. 프롬프트의 "Never call the same tool with the same arguments twice in a row" 가 무시됨. `prompts.build_messages` 가 user message 에 **직전 step 의 tool 호출 한 건만** 노출 (`Last tool call:`) — LLM 은 step 0 과 step 2 가 동일하다는 걸 가시적으로 못 본다.

기대: LLM 이 이전 호출 전체를 보고 같은 (tool, args) 를 반복하지 않음. 한두 번 retrieve 후 자료 요약 final_answer 또는 "좀 더 구체적인 질문 부탁드린다" 류 응답.

## 핵심 설계 결정

### 1. retrieval 관련성 마커 — windowing 매치와 분리된 strict 정책 (P2-1 보강)

`_focus_window` 가 미매치 fallback 으로 떨어졌는지 호출부에서 정확히 알기가 어려움 + **windowing 의 매치는 관련성 판정에 너무 느슨함**. 7-3 의 `_focus_window` 는 query 토큰 중 가장 빠른 첫 매치를 윈도우 중심으로 잡는데, 이 정책 그대로 관련성 판정에 쓰면 일반 토큰 ("비교", "차이", "얼마") 만 매치돼도 "관련 있음" 으로 잘못 판단. 예: query "우주여행 비교" 에서 "우주여행" 미매치라도 "비교" 가 청크 어딘가에 있으면 마커 안 붙음 → Defect 1 그대로.

해법: **windowing match 와 relevance match 를 분리**.

```python
# 의문/연산/비교 류 — query 의도 표현일 뿐 도메인 정보가 없는 토큰.
# 매치되어도 관련성 신호로는 부족 — relevance 판정에서 제외.
_LOW_SIGNAL_TOKENS = frozenset({
    # 비교/연산 의도
    '비교', '차이', '차이점', '대비',
    # 의문/요청
    '얼마', '얼마나', '뭐야', '뭔가요', '어떤', '어떻게', '있나', '있나요',
    '알려', '알려줘', '말해', '말해줘',
    # 일반 지시
    '관련', '대해', '대한',
    # 영문 일반어
    'compare', 'difference', 'about', 'what', 'how',
})


def _earliest_match(content: str, query: str) -> int:
    """**Windowing 용** 매치 위치. 7-3 _focus_window 의 토큰 우선순위 정책 그대로 —
    모든 토큰 (low-signal 포함) 후보. 미매치는 -1.

    이게 _focus_window 의 자리 잡기에 쓰이는 매치라 low-signal 토큰도 매치 대상.
    예: query '우주여행 비교' 에서 '우주여행' 매치 안 되고 '비교' 가 있으면 그 위치
    주변으로 윈도우. 관련성 신호로는 약하지만 windowing 정확성은 유지.
    """
    if not content or not query:
        return -1
    tokens = _tokenize_query(query)
    if not tokens:
        return -1
    lower = content.lower()
    for token in tokens:
        idx = lower.find(token.lower())
        if idx >= 0:
            return idx
    return -1


def _has_meaningful_match(content: str, query: str) -> bool:
    """**Relevance 마커 용** strict 판정 — 가장 긴 의미 토큰 (또는 동률 tier) 매치 요구.

    1) low-signal 토큰 (`비교`, `얼마` 등) 제외.
    2) 남은 의미 토큰 중 **가장 긴 길이의 토큰들** (max_len 같으면 tier 전체) 만
       매치 후보로 사용. 그 중 하나라도 청크에 있으면 True.

    이게 더 느슨한 정책 ('의미 토큰 하나라도 매치') 보다 엄격한 이유는, 짧은
    의미 토큰 (예: `비용`) 이 다른 도메인 청크에 우연히 들어있어도 정작 핵심
    도메인 명사 (`우주여행`) 가 미매치면 False relevant 가 되어 무한 retrieve
    회로가 남기 때문 (Defect 1 의 query 변형 `우주여행 비교` / `우주여행 비용
    비교` 등에서 발생). 가장 긴 의미 토큰이 도메인 키워드일 확률이 가장 높다는
    휴리스틱으로 strict 처리.

    동률 (max_len 같은 토큰 여러 개) 일 때는 그 중 하나라도 매치되면 OK — 입력
    토큰 순서 영향 최소화 (`결혼 휴가` 처럼 둘 다 2자인 query 에서 한쪽만 청크에
    있어도 관련 있음으로 인정).

    **알려진 한계 (tier-OR limitation)**: 같은 max_len 동률 토큰이 여러 개고 그 중
    하나가 일반 도메인 단어일 때 false relevant 발생. 예: query `우주여행 프로그램
    비교` 의 longest tier=[`우주여행`(4), `프로그램`(4)]. `프로그램` 이 다른 도메인
    문서 (휴가 프로그램 / 복지 프로그램 등) 에 우연히 있으면 True 로 판정돼 마커
    안 붙음. 이 케이스는 §Risk 표에 명시적으로 두고, 본 PR 의 Part 3 보강
    (`MAX_LOW_RELEVANCE_RETRIEVES=3` 누적 가드 + `_decide_termination` 우선순위
    재배치) 이 보완 방어선 역할 — LLM 이 서로 다른 query 변형으로 핑퐁하더라도
    `우주여행` 미매치 step 이 누적 3 도달하면 `low_relevance_retrieve_count` 누적
    가드가 max_iter 보다 먼저 NOT_FOUND 로 종료시킨다.

    query 가 모두 low-signal 이면 (예: '비교', '알려줘만') query 자체가 정보량
    부족이라고 보고 False — final_answer 로 query 다듬기 부탁드리는 게 정도.
    """
    if not content or not query:
        return False
    tokens = _tokenize_query(query)                    # already sorted len desc
    meaningful = [t for t in tokens if t.lower() not in _LOW_SIGNAL_TOKENS]
    if not meaningful:
        return False
    max_len = len(meaningful[0])
    longest_tier = [t for t in meaningful if len(t) == max_len]
    lower = content.lower()
    for token in longest_tier:
        if lower.find(token.lower()) >= 0:
            return True
    return False
```

### 2. recent tool calls 노출

현재 `prompts.build_messages` 의 user message:
```
Last tool call: retrieve_documents({'query': '...'})
```

직전 한 건만 보임. step 0 / step 2 가 같은 args 인 경우 step 3 시점에서 LLM 은 step 2 만 보고 step 0 의 존재를 모름 → 핑퐁.

변경:
```
Recent tool calls (last 5):
  1. retrieve_documents({'query': '복리후생 규정 비교'})
  2. retrieve_documents({'query': '복리후생 규정 세부사항'})
  3. retrieve_documents({'query': '복리후생 규정 비교'})
Do NOT repeat any of the above (tool, arguments) combinations.
```

K=5 결정 근거:
- `MAX_ITERATIONS=6` 이라 한 턴 최대 6 step. 5 면 거의 전체 history 노출.
- args 가 dict 라 한 줄당 50~100자 정도. 5 줄 ≈ 500자. catalog + observations + 이 리스트 합쳐도 user message 가 토큰 수천 단위로 폭주하지 않음.
- 부족 시 (예: 첫 step 에 tool_calls 0건) "Recent tool calls (none yet)." 로.

### 3. 프롬프트 두 군데 강화

### 4. Runtime guard — 세 갈래 (P2-1 + P2-2 보강)

세 갈래 보강 (Scope / 구현 순서 / 회귀 가드 모두 이 세 분류로 정렬):

**(part 1) 동일 (tool, args) 호출 차단** — LLM 의 핑퐁 패턴 대응. Defect 2 (`복리후생 규정 비교` 등) 정조준.

**(part 2) all-low-relevance / 0-hit retrieve 를 failure 처리** — `Tool.failure_check` + retrieve 의 `_retrieve_failure_check` 로 의미 없는 retrieve 결과를 `Observation.is_failure=True, failure_kind='low_relevance'` 로 신호화. LLM 이 query 변형으로 동일 args 차단을 우회해도 누적 가드의 카운트 단위가 됨.

**(part 3) low-relevance 누적 가드 + 우선순위 재배치** — `MAX_LOW_RELEVANCE_RETRIEVES=3` 누적 카운터를 `_decide_termination` 의 **최우선** (max_iter 보다 앞) 으로 두어 P2-1 의 마지막-step-동시-도달 케이스까지 NOT_FOUND 로 우선 종료. 진짜 알고리즘적 보장.

#### Part 1: 동일 (tool, args) 호출 차단

7-2 / 7-3 에서도 LLM 이 프롬프트의 "Never repeat ..." 를 무시한 사례가 있었음. recent tool calls 가시성 + No call repetition 가이드만으론 LLM 의 협조에 의존하는 정책이라 보장 부족.

해법: `react.py` 에서 LLM 이 동일 (tool, args) 호출을 시도해도 **실제 callable 호출을 차단** + 실패 Observation 만 누적. 같은 데이터로 같은 도구 호출은 deterministic 이라 2회째부터는 의미가 없음.

```python
# in react.py, after parsing action:
if isinstance(arguments, Mapping):
    # Phase 7-4: 같은 (tool, args) 가 이미 호출됐으면 실제 실행 차단.
    # 도구는 deterministic 이라 같은 인자면 같은 결과 — retry 가 의미 없음.
    if state.repeated_call_count(tool_name, arguments) >= 1:
        logger.info(
            'agent step %d: tool=%r 동일 호출 차단 (이전에 같은 인자로 호출됨)',
            state.iteration_count, tool_name,
        )
        state.add_observation(
            tool=tool_name,
            summary=(
                '이전과 같은 인자로 이미 호출했습니다. '
                '다른 query/tool 또는 final_answer 를 선택하세요.'
            ),
            is_failure=True,
            failure_kind='repeated_call',          # Phase 7-4: low_relevance 와 분리.
        )
        state.iteration_count += 1
    else:
        state.record_tool_call(tool_name, arguments)
        obs = agent_tools.call(tool_name, arguments)
        # ... (기존 동작)
```

이 가드와 `MAX_CONSECUTIVE_FAILURES=3` 의 조합으로 핑퐁 시나리오 (A → B → A → B → A) 가 다음과 같이 진행:

| step | LLM 시도 | runtime 행동 | consecutive_failures |
|---|---|---|---|
| 0 | A | 정상 실행 | 0 |
| 1 | B | 정상 실행 | 0 |
| 2 | A (반복) | **차단 + failure** | 1 |
| 3 | B (반복) | **차단 + failure** | 2 |
| 4 | A (반복) | **차단 + failure** | **3 → NO_MORE_USEFUL_TOOLS → NOT_FOUND** |

5 step 안에 종료 (현재 7-3 의 max 6 step). 같은 도구의 실제 retrieval / DB / LLM 비용은 정확히 한 번씩만 — 추가 0.

#### Part 2: `Tool.failure_check` 추가 + retrieve 의 all-low-relevance failure

LLM 이 핑퐁이 아니라 query 변형으로 회로를 만드는 케이스 (Defect 1). 매 step 의 `arguments` 가 다르니 Part 1 의 `repeated_call_count` 가드는 발동 안 함. 그래도 모든 retrieve 결과가 low-relevance 면 그 자체로 진전 없음 — failure 로 카운트해야 함.

`Tool` dataclass 에 optional `failure_check` 필드 추가:

```python
@dataclass(frozen=True)
class Tool:
    name: str
    description: str
    input_schema: Optional[Mapping[str, FieldSpec]]
    callable: Callable[[Mapping[str, Any]], Any]
    summarize: Callable[[Any], str]
    # Phase 7-4: callable 이 정상 반환했지만 의미상 실패로 처리할지 판정. None 이면 항상 success.
    failure_check: Optional[Callable[[Any], bool]] = None
```

`tools.call` 흐름 (예외 계약 + failure_kind 세팅 명시):
```python
result = tool.callable(arguments)
summary = tool.summarize(result)
is_failure = False
failure_kind: Optional[str] = None
if tool.failure_check is not None:
    try:
        if tool.failure_check(result):
            is_failure = True
            failure_kind = 'low_relevance'         # 누적 가드가 카운트할 종류.
    except Exception as exc:                                       # noqa: BLE001
        # failure_check 자체 버그가 자기충족적 종료 spiral 만들지 않게 — 소프트
        # 시그널이라 예외 시 not-failure 로 폴백 + 운영 가시성용 warning.
        logger.warning('failure_check error for tool %r: %s', tool.name, exc)
return Observation(
    tool=name, summary=summary,
    is_failure=is_failure, failure_kind=failure_kind,
)
```

(callable 예외는 별도 분기에서 `failure_kind='callable_error'`, schema 검증 실패는 `'schema_invalid'`, unknown tool 은 `'unknown_tool'` 등 동일 패턴으로 세팅 — 모두 `'low_relevance'` 와 분리되어 누적 가드에 카운트되지 않음.)

retrieve 에 등록:
```python
def _retrieve_failure_check(result):
    """retrieve 가 진전 없음 신호 — consecutive / 누적 failure 카운터용.

    P3 보강: 0건도 True (failure). "no useful evidence" 라는 의미가
    all-low-relevance 와 동일하므로 같은 신호로 통합.
    """
    query = (result or {}).get('query') or ''
    hits = (result or {}).get('hits') or []
    if not hits:
        return True  # 0건도 진전 없음으로 동일 취급 (P3).
    return not any(_has_meaningful_match(h.content, query) for h in hits)
```

#### Part 3: 누적 카운터 가드 + 우선순위 재배치 — UPSTREAM_ERROR 미도달 진짜 보장

Part 2 의 `failure_check` 만으로는 LLM 이 query 변형 + tier-OR false positive 가 끼어들면 consecutive_failures 가 리셋돼 max_iter=6 도달 가능. 누적 카운터 + 우선순위 재배치로 강화.

**Observation 에 failure_kind 분리** (P2-2 의 첫 번째 지적):

```python
@dataclass(frozen=True)
class Observation:
    tool: str
    summary: str
    is_failure: bool = False
    # Phase 7-4: failure 의 원인 종류. 'low_relevance' 만 누적 가드 카운트.
    # 'callable_error' / 'schema_invalid' / 'repeated_call' 등은 별도 분류.
    failure_kind: Optional[str] = None
```

`tools.call` 가 종류별로 세팅:
- `failure_check True` → `failure_kind='low_relevance'`
- callable 예외 → `'callable_error'`
- schema validation fail → `'schema_invalid'`
- unknown tool → `'unknown_tool'`
- (summarize 예외는 7-1 정책 그대로 — is_failure=False + 폴백 summary, failure_kind=None. 본 Phase 범위 밖.)

`react.py` 가 직접 추가하는 obs 도 종류 명시:
- `arguments` 가 non-Mapping → `'invalid_args'`
- 동일 (tool, args) call-block → `'repeated_call'`

**`AgentState`** (실행 오류와 자료 없음 분리):
```python
def low_relevance_retrieve_count(self) -> int:
    """retrieve_documents 의 'low_relevance' failure 만 누적 카운트.

    'callable_error' / 'schema_invalid' / 'repeated_call' 등 다른 종류 failure 는
    제외 — "자료 없음" 신호와 "실행 오류" 신호가 섞이지 않게 (P2-2 의 첫 번째 지적).
    """
    return sum(
        1 for obs in self.observations
        if obs.tool == 'retrieve_documents'
        and obs.failure_kind == 'low_relevance'
    )
```

**`react.py` — 우선순위 재배치** (P2-1 의 핵심 보강):
```python
MAX_LOW_RELEVANCE_RETRIEVES = 3

def _decide_termination(state, max_iterations):
    # 우선순위:
    # 1) low_relevance 누적 가드 — 마지막 step 에서 max_iter 와 동시 도달해도 NOT_FOUND 우선.
    if state.low_relevance_retrieve_count() >= MAX_LOW_RELEVANCE_RETRIEVES:
        return AgentTermination.NO_MORE_USEFUL_TOOLS
    # 2) max_iter 안전판.
    if state.iteration_count >= max_iterations:
        return AgentTermination.MAX_ITERATIONS_EXCEEDED
    # 3) 연속 실패.
    if state.consecutive_failures() >= MAX_CONSECUTIVE_FAILURES:
        return AgentTermination.NO_MORE_USEFUL_TOOLS
    # 4) 동일 호출 반복 (Part 1 차단으로 사실상 도달 불가, 하위 호환성).
    if state.tool_calls:
        last = state.tool_calls[-1]
        if state.repeated_call_count(last.name, last.arguments) >= MAX_REPEATED_CALL:
            return AgentTermination.NO_MORE_USEFUL_TOOLS
    return None
```

**진짜 보장 — 알고리즘 검증**:

case A — 모든 retrieve 가 low-relevance:
- step 0~2: 모두 fail → low_rel=3, consecutive=3
- iter=3 도달 직후 _decide_termination: low_rel >= 3 fires → NO_MORE_USEFUL_TOOLS → NOT_FOUND ✓

case B — tier-OR false positive 교차:
- step 0: fail (low_rel=1)
- step 1: false-pos (low_rel=1, consecutive=0)
- step 2: fail (low_rel=2)
- step 3: false-pos (low_rel=2, consecutive=0)
- step 4: fail (**low_rel=3 → fires**) → NOT_FOUND ✓

case C — 마지막 step 에서 동시 도달 (P2-1 의 직접적 케이스):
- step 0~1: fail/fail (low_rel=2)
- step 2~4: false-pos/false-pos/false-pos (low_rel=2)
- step 5: fail (low_rel=3, iter=6 = max_iter 동시)
- _decide_termination 우선순위: **low_rel 가드 먼저** → NO_MORE_USEFUL_TOOLS → NOT_FOUND ✓ (max_iter 가 먼저 잡지 않음)

case D — LLM 이 모든 step 에서 false-pos 만:
- step 0~5: 6번 모두 false-pos (성공) → low_rel=0, iter=6
- _decide_termination: low_rel < 3 → max_iter fires → MAX_ITERATIONS_EXCEEDED → UPSTREAM_ERROR
- 이 케이스는 의도된 UPSTREAM_ERROR — agent 가 6 step 내내 retrieve 성공했는데도 final_answer 안 함. 진짜 LLM 결함이라 운영적으로도 UPSTREAM_ERROR 가 맞음.

즉 보장은 **"retrieve 가 의미 없는 결과를 3회 이상 반환하는 시나리오에서 UPSTREAM_ERROR 미도달"** — Defect 1 의 본 케이스 정조준. 이건 LLM 행동 / tier-OR false positive 와 무관하게 알고리즘적으로 성립.

**Tool API 변경 영향**:
- `failure_check=None` 기본값이라 기존 Tool() 호출 (3 도구 모두) 호환.
- `tools.call` 의 호출 흐름은 try/except 포함 약 5 줄 추가.
- 다른 도구 (`find_canonical_qa`, `run_workflow`) 는 failure_check 미지정 → 기존 동작 그대로.

### 5. `MAX_REPEATED_CALL=3` 은 그대로 두되 사실상 dead-code

새 runtime guard 가 ≥1 회에서 차단하므로 같은 (tool, args) 의 `tool_calls` 누적은 사실상 1 을 못 넘김. 즉 `MAX_REPEATED_CALL=3` 가드는 도달 불가능 — runtime block 으로 대체된 셈. 상수와 가드 자체는 하위 호환성 / `_decide_termination` 시그니처 안정성 위해 그대로 유지.

대안 (미채택): `MAX_REPEATED_CALL=3` 도 같이 제거. 코드 깔끔해지지만 `_decide_termination` 변경 + state.repeated_call_count 사용처 정리가 따라옴 — 별 PR 의 일.

### 6. 정책 결합 효과

| LLM 행동 | 7-3 결과 | 7-4 결과 |
|---|---|---|
| 관련 없는 query 무한 retrieve (시나리오 3) | max_iter → UPSTREAM_ERROR | (a) `[관련성 낮음]` 마커 보고 1~2 step 내 final_answer (status=OK + "자료 못 찾았다") / (b) LLM 이 query 변형 회로 → `failure_check` + 누적 가드 (`MAX_LOW_RELEVANCE_RETRIEVES=3`) → 5 step 안에 NOT_FOUND. **low_relevance 가 3회+ 발생하는 시나리오에서 max_iter 도달 알고리즘적 차단** (tier-OR false positive 끼어들거나 마지막 step 동시 도달 모두). 단 LLM 이 6 step 모두 false positive 받는 진짜 LLM 결함 케이스는 UPSTREAM_ERROR — DoD 의 같은 예외 |
| 같은 (tool, args) 핑퐁 (시나리오 4) | max_repeated → NOT_FOUND | runtime guard 가 2회째 차단 → 누적 가드 / consecutive_failures 누적 → NOT_FOUND. 또는 LLM 이 마커/recent tool calls 보고 final_answer 로 정리 |
| 정상 비교 (회귀 0) | OK + 비교 답변 | OK + 비교 답변 |
| 정상 retry (LLM 이 같은 args 로 한 번 더 시도) | 정상 | 1회 정상, 2회째 차단 — consecutive_failures=1 누적이지만 다음 step 에서 다른 시도하면 카운트 리셋 |
