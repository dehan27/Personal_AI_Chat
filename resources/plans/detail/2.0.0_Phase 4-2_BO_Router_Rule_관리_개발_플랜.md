# 2.0.0 Phase 4-2 — BO Router Rule 관리 개발 플랜

## Context

Phase 4-1 이 `chat/services/question_router.py` 의 코드 상수 기반으로 `single_shot / workflow / agent` 3분류를 살려놨지만, 키워드가 코드에 박혀 있어 새 질문 패턴이 등장할 때마다 재배포가 필요하다. Phase 4-2 는 이 위에 **운영자가 BO 에서 rule 을 CRUD 하는 DB 레이어**를 얹어 override 가능하게 한다.

구조는 설계 §5-8 기준:

```
1) DB RouterRule 조회 (enabled=True, priority DESC)
      매치 있음 → 해당 route
2) 없으면 → 코드의 WORKFLOW_KEYWORDS / AGENT_KEYWORDS (Phase 4-1 fallback 유지)
3) 그래도 없으면 → 'single_shot' (default)
```

**코드 상수는 영구 보존**되어 "기본 동작" 역할을 맡고, DB rule 은 운영 중 조정하는 override 계층이 된다. BO 가 비어있어도 서비스는 Phase 4-1 상태 그대로 동작하므로 롤백이 쉽다.

핵심 제약:
- 사용자 응답(JSON / 화면) **회귀 0**
- `chat/services/single_shot/__init__.py` 공통 규칙 유지
- Phase 5~7 이 들어오기 전에 rule 관리 기반을 마련해 운영 조정이 선제 가능하도록

## 핵심 설계 결정

### 1. RouterRule 모델 필드
설계 §5-4 권장값 그대로. `Route` / `MatchType` 은 TextChoices, route 문자열은 `chat.graph.routes` 상수와 동일한 리터럴 사용.

```python
class RouterRule(models.Model):
    class Route(models.TextChoices):
        SINGLE_SHOT = 'single_shot', 'single-shot'
        WORKFLOW = 'workflow', 'workflow'
        AGENT = 'agent', 'agent'

    class MatchType(models.TextChoices):
        CONTAINS = 'contains', '포함 (contains)'
        # regex / exact 는 Phase 4-2 확장 후보 — choices 에 넣지 않음

    name = models.CharField(max_length=100, help_text='운영자 식별용 이름')
    route = models.CharField(max_length=20, choices=Route.choices)
    match_type = models.CharField(max_length=20, choices=MatchType.choices, default=MatchType.CONTAINS)
    pattern = models.CharField(max_length=256, help_text='매칭 키워드 / 패턴')
    priority = models.IntegerField(default=100, help_text='숫자가 클수록 먼저 평가')
    enabled = models.BooleanField(default=True)
    description = models.TextField(blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ['-priority', '-updated_at']
```

제약/인덱스:
- `unique_together` 없음 — 동일 pattern 도 여러 rule 로 두고 싶을 수 있음 (다른 route, priority 로)
- 기본 ordering 이 priority DESC 라 router 가 별도 정렬 없이 바로 순회 가능

### 2. `route_question` 우선순위 변경
`chat/services/question_router.py` 에 `_match_db_rules(question) -> Optional[RouteDecision]` 추가. DB 조회는 lazy import 로 (`from chat.models import RouterRule` 를 함수 안에서) 앱 로드 순서 문제 회피.

```python
def route_question(question: str) -> RouteDecision:
    db_decision = _match_db_rules(question)
    if db_decision is not None:
        return db_decision

    # (기존 Phase 4-1 keyword fallback + default single_shot)
```

`_match_db_rules` 반환 시 `reason='db_rule:<name>'`, `matched_rules=[rule.pattern]`. 로그·BO 관측에서 어느 rule 이 터졌는지 추적 가능.

매 요청 DB 조회. 설계 §5-9 옵션 1. 초기 rule 수 <100 수준이면 부하 문제 없음.

README / dev log 에 "기본 동작은 코드 키워드, BO 는 override 용" 명시.

### 7. ROUTE_* 상수 재사용
모델 `Route.choices` 의 문자열 값은 `chat.graph.routes.ROUTE_SINGLE_SHOT` 등과 **동일한 리터럴**. 두 곳이 어긋나지 않도록 모델 파일에서 `from chat.graph.routes import ...` 로 검증하는 대신, TestChoices 의 값으로 직접 `'single_shot'` 등을 쓰되 docstring 에 "chat.graph.routes 와 동기화" 라고 경고. (Django 모델 정의 시점에 다른 앱 상수를 import 하는 건 app load 이슈 가능성 있어 보수적으로.)
