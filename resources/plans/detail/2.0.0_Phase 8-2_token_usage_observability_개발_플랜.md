## 핵심 설계 결정

설계 §6-4 의 4 가지 질문 + §6-5 권장 방향에 대한 본 plan 의 답:

### 1. `purpose` 의 타입 — **CharField + 코드 상수** (DB enum 아님)

설계 §6-4 첫 질문에 대한 답.

- **DB 측**: `CharField(max_length=32, default='unknown')` — 자유 문자열로 두고 마이그레이션 / 다른 환경 (테스트 fixture, 데이터 import) 에서도 깨지지 않게.
- **코드 측**: `chat/services/token_purpose.py` 의 상수가 single source of truth. `ALL_PURPOSES` 가 모든 알려진 값.
- **방어망**: `validate_purpose(value)` 가 record 진입 시 알 수 없는 값을 'unknown' 으로 절감 + warning 로그. **호출부 오타가 데이터 오염으로 직결되지 않음**.

trade-off:
- ✓ DB 마이그레이션 단순 (default + db_index 만 추가).
- ✓ Phase 8-3 / 후속이 새 purpose 추가 시 코드 한 줄만.
- ✓ 외부 데이터 import 가 모르는 purpose 를 가져와도 전체 시스템이 깨지지 않음.
- ✗ DB 자체로는 "이 값이 valid 한가" 보장 X — 하지만 record_token_usage 가 진입점이라 **모든 신규 row 는 validate_purpose 를 통과** → 실질 보장.

### 2. Migration 정책 — **AddField + default='unknown'**, 별도 backfill RunPython 없음

설계 §6-4 두번째·세번째 질문에 대한 답.

- Django `AddField(default='unknown')` 가 기존 row 에 자동 적용 — 단 한 번의 ALTER TABLE.
- 본 plan 시점에 production / dev / test DB 모든 기존 TokenUsage row 가 `purpose='unknown'` 으로 즉시 분류됨.
- 별도 RunPython backfill 불필요 — 데이터 마이그레이션 위험 0.
- 향후 BO 대시보드가 purpose 별 분해를 보여줄 때 "8-2 머지 이전" 데이터는 모두 'unknown' bucket 에 모이는 게 정확한 의미 (실제로 그 시점에는 purpose 정보가 없었음).

테스트는 신규 row 작성 시 default 가 'unknown' 으로 떨어지는지 회귀 가드 1개로 충분.

### 3. `record_token_usage` 시그니처 — **keyword-only 추가, positional 시그니처 유지**

```python
def record_token_usage(model: str, usage: Any, *, purpose: str = PURPOSE_UNKNOWN) -> None:
```

이 형태의 의미:

- 기존 모든 호출자 (`record_token_usage(rewriter_model, rewriter_usage)`) 는 깨지지 않음 — purpose 가 default로 'unknown' 이 됨.
- 7 호출 사이트는 본 plan 안에서 명시 전달로 전환 → 머지 후 모든 신규 row 는 정확한 purpose.
- **migration 시점과 호출부 변경 시점 사이에 잠시 'unknown' row 가 쌓여도 문제 없음** — 호출부 변경이 다 끝난 시점부터 'unknown' 은 의도된 케이스 (예: 외부 호출자 / 새 경로 누락) 만 남음.

이 점진성이 핵심: migration 만 머지하고 호출부는 별 PR 로 가도 회귀 0. 다만 본 plan 은 한 PR 안에서 다 끝내므로 잠시 윈도우가 매우 짧음.

### 4. agent ReAct 의 `agent_step` vs `agent_final` 구분 — **call-site 분기**

설계 §6-3 의 후보 중 `agent_step` / `agent_final` 두 값을 구분할지가 디테일.

옵션:
- (A) **모두 `agent_step` 으로 통일** — 단순. agent reasoning + final_answer 비용을 한 bucket 에서 봄.
- (B) **call-site 분기** — `react.py` 가 LLM 응답 파싱 후 action == 'final_answer' 면 `PURPOSE_AGENT_FINAL`, 아니면 `PURPOSE_AGENT_STEP`.

본 plan 은 **(B)** 채택:

- 운영 관점에서 "agent 가 답 만들 때 든 비용" vs "탐색 step 에 든 비용" 을 분리해서 보는 가치가 있음. 후자가 비대해지면 max_iterations / 도구 결정 정책 검토.
- 구현 비용 미미 — `record_token_usage` 호출부에 `purpose=` 1줄 분기.
- 재사용성: 후속 Phase 가 "iteration 별 비용 시계열" 보고 싶을 때 base.

trade-off:
- ✓ 비용 분석 정밀도 ↑.
- ✗ 약간의 fragility — `final_answer` action parsing 실패 시 `agent_step` 로 떨어지지만, 이 경우 `_llm` parse 실패 path 라 token 자체가 적어 영향 미미.

구현 위치: `react.py` 의 `record_token_usage(model, usage)` 호출 직전 `action = _parse_action(raw)` 결과를 보고 분기. **action parsing 이 record 이전에 이미 일어나는지 확인** — 현재 코드는 `record_token_usage(model, usage)` 가 `_parse_action` 호출 이전에 있어 분기 시점 조정이 필요. §구현 순서 Step 4 에서 처리.

### 5. `query_rewriter` 는 3 호출 사이트 공통 purpose — **하나로 통합**

`single_shot / workflow / agent` 세 경로 모두 `rewrite_query_with_history` 를 호출 → rewriter LLM 사용량이 발생. 세 경로의 purpose 를 어떻게 분류할지:

옵션:
- (A) **모두 `query_rewriter`** — 같은 함수, 같은 비용 패턴.
- (B) `query_rewriter_single_shot` / `query_rewriter_workflow` / `query_rewriter_agent` 분리.

본 plan 은 **(A)**:

- rewriter 의 비용 패턴은 호출 경로와 거의 무관 (같은 모델 / 같은 history slice / 같은 프롬프트).
- 분류 늘리면 후속 분석에 noise 만 추가.
- 정말 경로별 분해가 필요해지면 후속 Phase 에서 `purpose` 를 변경하지 않고 별 차원 (예: `route` 메타) 으로 추가 — purpose 자체는 의미 단위로 유지.

### 사용자-facing smoke

8-2 는 운영자 표면이라 사용자-facing 변화 없음. 검증은 **DB 행 직접 확인** + **신규 row 만 분리해서 보기**.

먼저 머지 직후 base 시점을 잡아 둔다 (이후 비교용):

```bash
docker compose exec -T web python manage.py shell -c "
from chat.models import TokenUsage
print('base_max_id =', TokenUsage.objects.aggregate(m=__import__('django.db.models', fromlist=['Max']).Max('id'))['m'] or 0)
"
```

`base_max_id` 를 기록한 뒤, 아래 **5 시나리오 시퀀스** 를 한 세션에서 차례로 실행:

| # | 시나리오 (브라우저) | 기대 라우트 | 발생할 신규 purpose |
|---|---|---|---|
| 1 | `경조사 규정 알려줘` (첫 질문, history 없음) | single_shot | `single_shot_answer` |
| 2 | `위 규정에서 가장 비싼 항목은?` (후속, history 있음) | single_shot | `query_rewriter`<sup>※1</sup> + `single_shot_answer` |
| 3 | `2025년 1월 1일부터 100일 후 날짜는?` | workflow (`date_calculation`) | `workflow_extractor` <sup>※2</sup> |
| 4 | `복리후생 규정에서 본인 결혼 경조금은?` | workflow (`table_lookup`) | `query_rewriter`<sup>※2</sup> + `workflow_extractor` + `workflow_table_lookup` |
| 5 | `본인 결혼 경조금이랑 자녀 결혼 경조금 비교해줘` | agent | `query_rewriter`<sup>※1</sup> + `agent_step` × N + `agent_final` × 1 |

<sup>※1</sup> single_shot / agent 경로의 rewriter 는 **history 만** 으로 결정 — history 가 있으면 호출. 위 시퀀스가 한 세션이라 시나리오 2 / 5 는 history 누적 → 호출됨.

<sup>※2</sup> workflow 경로의 rewriter 는 `workflow_node._schema_needs_retrieval` 휴리스틱으로 추가 게이트 — `input_schema` 에 `'text'` 타입 필드가 있는 workflow 만 호출 (`graph/nodes/workflow.py:60`). `table_lookup` 은 `'query': type='text'` 라 호출 / `date_calculation` (date/date/enum) 과 `amount_calculation` (number_list/enum) 은 text 필드 없음 → **history 가 있어도 호출 0**. 따라서 시나리오 3 은 history 가 있어도 `query_rewriter` 가 발생하지 않고 `workflow_extractor` 만 기록됨. 시나리오 4 는 text schema 라 rewriter 호출됨.

신규 row 만 추출해 분포 확인:

```bash
docker compose exec -T web python manage.py shell -c "
from chat.models import TokenUsage
from collections import Counter
BASE = <base_max_id>   # 위에서 기록한 값
new_rows = TokenUsage.objects.filter(id__gt=BASE)
c = Counter(new_rows.values_list('purpose', flat=True))
print('신규 row purpose 분포:', dict(c))
print('신규 row 개수:', new_rows.count())
"
```

**통과 기준** (P2 보강):

- ✓ **known 6종이 모두 신규 row 에 등장** — `single_shot_answer` / `query_rewriter` / `workflow_extractor` / `workflow_table_lookup` / `agent_step` / `agent_final`.
- ✓ **신규 row 의 `unknown` 카운트는 0** — 누락 호출이 없다는 시그널. (전체 DB 의 `unknown` 누계는 머지 이전 기록분이 있으면 양수일 수 있음 — 위 base 분리로 격리.)
- ✓ `agent_step` 카운트는 N (시나리오 5 의 ReAct 루프 step 수) ≥ 1, `agent_final` 은 정확히 1.

분포가 6종에 못 미치면 **누락 호출 사이트** 또는 **시나리오 라우팅 실패**. 후자는 RouterRule 점검 (`bo:router_rules` 페이지 — 시나리오 4 가 `table_lookup` 으로, 5 가 `agent` 로 매치되는 RouterRule 이 등록되어 있어야 함).
