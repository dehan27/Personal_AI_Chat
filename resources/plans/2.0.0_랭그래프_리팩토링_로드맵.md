# 2.0.0 랭그래프 리팩토링 로드맵

## 1. 목표

`2.0.0`의 최종 목표는 현재의 single-shot RAG 챗봇을 `LangGraph` 기반 구조로 리팩토링하고, 질문 처리 방식을 아래 3가지로 분류해 안정적으로 운영할 수 있게 만드는 것이다.

1. `single-shot`
   - 단순 문서 검색형 질문 처리
2. `workflow`
   - 정해진 절차와 계산이 있는 질문 처리
3. `agent`
   - ReAct 기반의 루프형 탐색/추론 질문 처리

핵심 원칙은 다음과 같다.

- 현재 동작 중인 single-shot 품질을 최대한 유지한다.
- 프롬프트와 실행 흐름을 코드에서 분리해 운영성을 높인다.
- 프레임워크가 담당할 orchestration과 서비스 고유 로직을 분리한다.
- 처음부터 agent를 크게 만들지 않고 `single-shot -> workflow -> agent` 순서로 확장한다.
- BO, 채팅, 향후 추가 화면을 포함한 모든 UI/UX 변경은 프로젝트 공용 HTML 가이드라인을 기준으로 한다.

---

## 2. 왜 이 순서로 가는가

바로 ReAct agent부터 붙이면 현재 구조를 크게 흔들 가능성이 높다. 따라서 먼저 `프롬프트 분리`, `LangGraph 진입`, `기존 single-shot 래핑`을 끝낸 뒤, 그 위에서 workflow와 agent를 추가하는 방식으로 진행한다.

이 순서를 따르면 다음 이점이 있다.

- 현재 서비스 기능을 잃지 않는다.
- LangGraph 도입 후에도 기존 검색/재정렬 로직을 재사용할 수 있다.
- 이후 workflow와 agent를 같은 실행 프레임 안에서 확장할 수 있다.
- 나중에 프롬프트 관리와 운영 관리 구조를 넓히기 쉽다.

---

## 3. 단계별 마일스톤

### Phase 0. 기준선 확보

목적:
- 리팩토링 전후 품질 비교 기준을 만든다.

할 일:
- 현재 single-shot 질문셋 정리
- 대표 질문 카테고리 정리
- 실패 사례/애매한 답변 사례 수집
- 최소 회귀 체크리스트 작성

산출물:
- 대표 질문셋
- 회귀 확인 체크리스트

완료 기준:
- 리팩토링 후 기존 동작이 깨졌는지 비교 가능한 상태

---

### Phase 1. Prompt 관리 분리

목적:
- 프롬프트를 코드 상수에서 분리하고, BO에서 수정 가능한 관리 기반을 확보한다.

할 일:
- `assets/prompts` 또는 이에 준하는 비공개 디렉토리 설계
- 시스템 프롬프트 외부 파일화
- no-source, router, workflow용 프롬프트 파일 구조 초안 설계
- 프롬프트 로더 계층 설계
- BO Prompt 관리 페이지 설계

권장 범위:
- 우선은 파일 기반
- BO에서는 허용된 prompt 파일만 수정 가능하도록 제한
- 나중에 DB 기반으로 확장 가능하도록 로더 인터페이스를 단순하게 유지

산출물:
- 프롬프트 디렉토리 구조
- Prompt Loader 설계 문서
- BO Prompt 관리 UI 설계 문서

완료 기준:
- 프롬프트가 코드 하드코딩에서 분리될 준비가 완료됨
- 운영자가 BO에서 prompt를 확인/수정할 수 있는 방향이 정의됨

---

### Phase 2. LangGraph 적용

목적:
- 현재 서비스 진입점을 LangGraph 기반 실행 구조로 전환한다.

할 일:
- LangGraph state 설계
- 최소 graph 진입 구조 설계
- `single_shot_node` 개념 정의
- 현재 `answer_question()`을 graph node에서 호출하는 래퍼 구조 설계

중요 포인트:
- 이 단계에서는 기존 single-shot 로직을 유지한다.
- 아직 workflow와 agent를 활성화하지 않는다.
- 기존 `answer_question()` 래핑은 최종 구조가 아니라 안전한 이관을 위한 임시 단계로 본다.

산출물:
- graph 진입 구조 문서
- state/schema 초안

완료 기준:
- 현재 single-shot이 LangGraph 프레임 안에서 돌 수 있는 구조가 정의됨

---

### Phase 3. 기존 코드 개편

목적:
- 현재 코드를 LangGraph 친화적으로 재배열한다.

할 일:
- orchestration 코드와 도메인 로직 분리
- 검색/재정렬/프롬프트 조립/응답 후처리 경계 재정리
- `chat/services` 내 책임 재배치
- 향후 `single-shot / workflow / agent` 확장을 고려한 디렉토리 구조 설계
- 초기 래핑으로 연결한 기존 코드 중, LangGraph가 더 잘 담당할 수 있는 부분 식별
- 프레임워크로 대체 가능한 직접 구현 영역을 점진적으로 치환
- 임시 래핑 코드와 중복 orchestration 코드 정리

권장 방향:
- 프레임워크가 나중에 대체할 부분은 얇게 유지
- 검색, 계산, 라우팅 기준, 도메인 규칙은 재사용 가능한 함수로 유지
- `처음엔 래핑 -> 이후 점진 치환 -> 최종 graph-native 구조 정착` 순서로 진행

산출물:
- 리팩토링 대상 목록
- 변경 후 패키지 구조 초안
- 프레임워크 치환 대상 목록

완료 기준:
- 이후 workflow/agent 추가 시 기존 파일이 과도하게 엉키지 않는 구조 확보
- 프레임워크가 더 잘 담당할 수 있는 영역이 직접 구현 코드에서 분리되기 시작한 상태

---

### Phase 4. 질문 라우팅 설계

목적:
- 최종 3분류 구조의 진입 조건을 명시한다.

할 일:
- `single-shot / workflow / agent` 라우팅 기준 정의
- 규칙 기반 1차 라우터 설계
- 애매한 질문 처리 원칙 정의
- 실패 시 상위 경로 승급 기준 정의

권장 원칙:
- 처음에는 규칙 기반으로 시작
- 필요 시 저비용 모델 분류를 보조적으로 고려

산출물:
- 라우터 조건표
- 예시 질문 분류표

완료 기준:
- 어떤 질문이 어떤 경로로 가는지 설명 가능한 상태

---

### Phase 5. Workflow Core 개발

목적:
- 도메인과 무관하게 재사용 가능한 workflow 공통 기능을 만든다.

예시 범위:
- 날짜 변환
- 기간 계산
- 덧셈/평균/합계
- 누락값 검사
- 값 정규화
- 계산 결과 포맷팅

핵심 원칙:
- 이 레이어는 특정 회사 규칙을 몰라야 한다.
- 순수 함수 중심으로 설계한다.

산출물:
- 공통 계산 유틸 목록
- workflow core 책임 정의

완료 기준:
- 여러 workflow에서 공통 재사용 가능한 기반이 준비됨

---

### Phase 6. Workflow Domain 로직 설계 및 개발

목적:
- 실제 업무형 질문을 처리하는 도메인별 workflow를 만든다.

예시 범위:
- 퇴직금
- 근속기간
- 연차 계산
- 수당/정산 관련 계산

핵심 원칙:
- core와 domain을 분리한다.
- 도메인 규칙과 공식은 별도 계층으로 관리한다.
- 가능한 한 정해진 절차형 workflow로 처리하고, 불필요하게 agent로 보내지 않는다.

산출물:
- 도메인 workflow 목록
- 각 workflow의 입력/출력/필수값 정의

완료 기준:
- 최소 1개 이상의 대표 workflow가 실제 동작 가능한 수준으로 설계됨

---

### Phase 7. Agent 개발

목적:
- 고정 workflow로 다루기 어려운 질문을 ReAct 기반 agent 경로로 처리한다.

예상 범위:
- 다중 검색
- 조건별 비교
- 예외 케이스 탐색
- 추가 조회가 필요한 열린 질문 처리

핵심 원칙:
- agent는 마지막 단계에 도입한다.
- 모든 복잡 질문을 agent로 보내지 않는다.
- workflow로 해결 가능한 것은 workflow에 남긴다.

산출물:
- agent 경로 설계 문서
- 최소 tool 목록
- loop 제한 정책

완료 기준:
- ReAct agent의 역할 범위가 workflow와 명확히 구분됨

---

### Phase 8. 2.0.0 안정화 및 배포 준비

목적:
- 최종 배포 전 품질과 구조를 고정한다.

할 일:
- 대표 질문 회귀 검증
- 3분류 라우팅 검증
- 프롬프트 파일 구조 점검
- 문서 정리
- 배포 체크리스트 작성

완료 기준:
- `single-shot / workflow / agent` 3분류가 모두 정의되고, 최소 동작 검증이 끝난 상태

---

## 4. 추천 최종 순서

현재 기준 추천 순서는 아래와 같다.

1. Phase 0. 기준선 확보
2. Phase 1. Prompt 관리 분리
3. Phase 2. LangGraph 적용
4. Phase 3. 기존 코드 개편
5. Phase 4. 질문 라우팅 설계
6. Phase 5. Workflow Core 개발
7. Phase 6. Workflow Domain 로직 설계 및 개발
8. Phase 7. Agent 개발
9. Phase 8. 2.0.0 안정화 및 배포 준비

즉, 처음 정리한 방향은 맞지만 아래 3가지를 보강한 버전으로 가는 것이 좋다.

- 시작 전에 `기준선 확보`
- workflow/agent 전에 `라우터 설계`
- 마지막에 `안정화/배포 준비`

---

## 5. 단계별 우선순위 요약

### 먼저 해야 하는 것

- Prompt 분리
- LangGraph 진입 구조
- 기존 single-shot 유지 상태에서 graph 래핑

### 그다음 해야 하는 것

- 코드 구조 재정리
- 라우팅 기준 확정
- workflow core 구축

### 마지막에 해야 하는 것

- domain workflow
- agent
- 2.0.0 안정화 및 배포

---

## 6. 2.0.0 완료 정의

`2.0.0`은 아래 조건을 만족할 때 완료로 본다.

- 현재 single-shot 기능이 LangGraph 기반 구조로 이관되었다.
- 프롬프트가 코드 하드코딩에서 분리되었다.
- 질문이 `single-shot / workflow / agent` 3분류로 라우팅된다.
- 최소 1개 이상의 대표 workflow가 동작한다.
- agent 경로의 역할과 범위가 명확히 정의되었다.
- 배포 전 회귀 검증 기준이 준비되었다.

---

## 7. 메모

- `LlamaIndex`는 당장 우선순위에서 제외한다.
- 문서 처리층은 현재 구현을 최대한 유지한다.
- `LangSmith`는 비용 이슈로 우선 제외하고, 운영 관측은 자체 로그/문서 기준으로 정리한다.
- 초기 목표는 `full agent 전환`이 아니라 `LangGraph 기반 구조 안착`이다.
