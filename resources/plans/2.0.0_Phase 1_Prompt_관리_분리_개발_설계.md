# 2.0.0 Phase 1 Prompt 관리 분리 개발 설계

## 1. 목적

Phase 1의 목적은 현재 코드 내부에 하드코딩되어 있는 프롬프트를 코드 밖의 비공개 파일로 분리하고, 이후 `LangGraph` 구조로 넘어가더라도 재사용 가능한 `Prompt Loader` 기반을 먼저 만드는 것이다.

이 단계의 핵심 목표는 아래 4가지다.

1. 프롬프트를 코드 상수에서 분리한다.
2. 프롬프트를 일관된 방식으로 읽는 로더 계층을 만든다.
3. BO에서 프롬프트를 확인하고 수정할 수 있는 운영 UI 기반을 만든다.
4. 이후 `single-shot / workflow / agent` 확장 시 같은 방식으로 프롬프트를 추가할 수 있게 한다.

---

## 2. 현재 상태

현재 프로젝트에서 프롬프트는 코드에 직접 들어 있다.

- 시스템 프롬프트: `chat/prompt/chat.py`
- 메시지 조립 시 사용: `chat/services/prompt_builder.py`
- 히스토리 초기값에서도 사용: `chat/services/history_service.py`

현재 구조의 문제는 다음과 같다.

- 프롬프트 변경이 곧 코드 변경이다.
- 운영 중 프롬프트 수정 시 재배포 단위로 묶이기 쉽다.
- 프롬프트가 늘어나면 코드 파일과 문자열 상수가 계속 비대해진다.
- `LangGraph` 도입 후 prompt 역할별 분리가 필요해질 때 구조가 버티기 어렵다.

---

## 3. 이번 Phase의 범위

이번 Phase에서는 아래까지만 한다.

- 시스템 프롬프트 파일 분리
- 현재 single-shot에서 사용하는 보조 지시문 파일 분리
- 프롬프트 로더 계층 설계 및 적용
- 설정값으로 프롬프트 디렉토리 경로를 주입할 수 있게 준비
- BO Prompt 관리 페이지 설계 및 추가
- 허용된 prompt 파일만 수정 가능한 안전장치 설계

이번 Phase에서 하지 않는 것:

- DB 기반 프롬프트 관리
- LangSmith 연동
- workflow/agent 프롬프트 실제 사용
- 프롬프트 실시간 핫리로드 고도화

즉, 이 단계는 `운영 가능한 파일 기반 Prompt 관리 구조 + BO 수정 UI 초석 마련`이 목표다.

## 4. 설계 원칙

### 4-1. 공개 자원과 비공개 프롬프트를 분리한다

현재 `resources/`는 `/media/`로 공개 서빙되는 경로다. 따라서 프롬프트는 여기에 두지 않는다.

권장 경로:

- 기본 개발 경로: `assets/prompts/`
- 운영 확장 경로: 환경변수 `PROMPTS_DIR`로 외부 경로 override

즉,

- 개발 기본값은 repo 내부 파일
- 운영에서는 필요 시 repo 밖 mounted path

구조로 간다.

### 4-2. 프롬프트는 코드에서 직접 읽지 않는다

각 서비스 파일이 직접 `open()`을 호출하지 않고, 중앙 `Prompt Loader`를 통해 읽는다.

이유:

- 경로 관리 일원화
- fallback 정책 일원화
- 캐시 정책 추가 가능
- 나중에 DB/외부 저장소로 바꾸기 쉬움

### 4-3. BO에서는 모든 파일을 수정하게 두지 않는다

BO에서 파일을 직접 수정하게 할 경우 가장 중요한 것은 범위 제한이다.

권장 원칙:

- `PROMPTS_DIR` 하위 전체를 자유 입력으로 수정하지 않는다.
- 미리 등록된 prompt 파일만 편집 가능하게 한다.
- 상대 경로는 서버 코드에서만 관리한다.

즉, BO는 `파일 탐색기`가 아니라 `허용된 Prompt 편집기`로 설계한다.

### 4-4. 프롬프트는 역할 단위로 나눈다

한 파일에 모든 문자열을 몰아넣지 않고 역할별로 쪼갠다.

예상 역할:

- system prompt
- no-source guard
- source instruction
- QA reference instruction
- 향후 router prompt
- 향후 workflow prompt
- 향후 agent tool prompt

### 4-5. 지금은 파일 기반이지만 인터페이스는 확장 가능하게 둔다

처음 구현은 파일 기반으로 충분하다. 다만 로더 인터페이스는 이후에도 재사용 가능해야 한다.

예:

- `load_text("chat/system.md")`
- `load_prompt("chat/no_sources.md")`

처럼 단순한 API로 고정한다.

## 5. 목표 디렉토리 구조

초기 구조는 아래를 권장한다.

```text
assets/
  prompts/
    chat/
      system.md
      source_instruction.md
      qa_instruction.md
      no_sources_guard.md
```

향후 확장 구조는 아래를 고려한다.

```text
assets/
  prompts/
    chat/
      system.md
      source_instruction.md
      qa_instruction.md
      no_sources_guard.md
    router/
      classify.md
    workflows/
      common.md
      retirement.md
    agents/
      react_system.md
```

현재 Phase에서는 `chat/`까지만 실제 사용 대상으로 본다.

### 5-1. BO Prompt 관리 화면 방향

현재 BO는 대시보드, 파일관리, Q&A 관리 중심이다. 여기에 `Prompt 관리` 메뉴를 추가하는 방향으로 본다.

UI/UX 원칙:

- BO Prompt 관리 화면도 프로젝트 공용 HTML UI/UX 가이드라인을 따른다.
- 이후 채팅 화면, BO 화면, 신규 운영 화면 모두 같은 가이드라인을 기준으로 구현한다.
- 가이드라인은 특정 앱 전용 템플릿이 아니라 프로젝트 공용 기준 자산으로 관리한다.

예상 메뉴:

- 대시보드
- 파일관리
- Q&A 관리
- Prompt 관리

Phase 1에서의 BO Prompt 관리 기능 범위:

- 편집 가능한 prompt 목록 조회
- prompt 본문 확인
- prompt 본문 수정 및 저장
- prompt별 간단 설명 표시

Phase 1에서 하지 않는 기능:

- prompt 버전 diff 비교
- rollback UI
- 다중 환경(dev/staging/prod) 분기 관리
- draft/publish 승인 플로우

즉, 처음에는 `간단한 파일 기반 Prompt 편집 UI`로 시작한다.

## 6. 설정 설계

`settings.py`에 프롬프트 경로 설정을 추가한다.

권장 방향:

- 기본값: `BASE_DIR / "assets" / "prompts"`
- override: `PROMPTS_DIR` 환경변수

예상 설정 개념:

```python
PROMPTS_DIR = Path(os.environ.get("PROMPTS_DIR", BASE_DIR / "assets" / "prompts"))
```

주의:

- repo 내부 경로만 쓰면 운영에서 재배포 없는 수정이 제한될 수 있다.
- 따라서 운영에서는 `PROMPTS_DIR`를 외부 마운트 경로로 줄 수 있게 해둔다.

## 7. Prompt Loader 설계

### 7-1. 역할

Prompt Loader는 다음을 담당한다.

- 프롬프트 파일 경로 계산
- UTF-8 텍스트 로드
- 공통 예외 처리
- 존재하지 않는 파일 처리 정책
- 필요 시 캐싱 확장 포인트 제공

### 7-2. 위치

권장 위치:

- `chat/services/prompt_loader.py`

이유:

- 현재 prompt 사용처가 chat 앱 중심
- LangGraph 도입 전 단계에서도 바로 적용 가능

### 7-3. 최소 API

권장 최소 API:

- `load_prompt(relative_path: str) -> str`

예:

- `load_prompt("chat/system.md")`
- `load_prompt("chat/no_sources_guard.md")`

### 7-4. 에러 정책

파일 누락은 조용히 삼키지 않는다.

권장 정책:

- 파일이 없으면 명시적 예외 발생
- 개발/운영 모두 빠르게 실패하도록 한다

이유:

- 프롬프트 누락 상태로 서비스가 애매하게 동작하는 것이 더 위험하다

### 7-5. 캐시 정책

Phase 1에서는 단순 구현으로 시작한다.

권장 방식:

- 프로세스 메모리 캐시 없이 매 호출 로드 또는
- 매우 얕은 in-process cache만 사용

초기 판단:

- 현재 요청량이 크지 않다면 구현 단순성을 우선한다
- 이후 필요 시 캐싱 추가

## 8. BO Prompt Registry 설계

BO에서 어떤 prompt를 보여주고 수정할지는 별도 registry로 관리하는 것이 좋다.

권장 이유:

- 허용된 파일만 수정 가능
- 제목/설명/정렬 순서 관리 가능
- 추후 workflow/agent prompt 추가 시 등록형으로 확장 가능

예상 개념:

- key
- title
- description
- relative_path
- editable 여부

예시:

```python
PROMPT_REGISTRY = [
    {
        "key": "chat_system",
        "title": "채팅 시스템 프롬프트",
        "description": "기본 말투와 응답 원칙",
        "relative_path": "chat/system.md",
        "editable": True,
    },
]
```

이 registry는 Phase 1에서는 코드 상수로 두는 것이 가장 단순하다.

## 9. BO Prompt 관리 페이지 설계

### 9-1. 예상 위치

권장 위치:

- `bo/views/prompts.py`
- `bo/templates/bo/prompts.html`
- `bo/templates/bo/prompts_edit.html`

라우팅 예시:

- `/bo/prompts/`
- `/bo/prompts/<key>/`
- `/bo/prompts/<key>/update/`

### 9-2. 화면 역할

목록 화면:

- 편집 가능한 prompt 목록 노출
- title / description / 파일 경로 표시
- 수정 화면으로 이동

수정 화면:

- prompt 본문 textarea 표시
- 저장 버튼
- 파일 경로와 주의사항 표시

화면 설계 기준:

- 레이아웃, 간격, 타이포그래피, 버튼 스타일, 폼 구성은 공용 UI 가이드라인을 기준으로 맞춘다.
- 새 화면을 만들 때는 기존 BO 패턴만 복제하는 것이 아니라 공용 가이드라인과의 일치 여부를 함께 본다.

### 9-3. 저장 정책

권장 정책:

- 저장 시 registry에 등록된 파일인지 확인
- `PROMPTS_DIR / relative_path`로만 저장
- 임의 path 입력 금지
- UTF-8 기준 저장

### 9-4. 보안/안전 원칙

- path traversal 차단
- registry 미등록 파일 접근 금지
- 빈 prompt 저장 방지 여부 정책 검토
- 운영 로그 또는 Django message로 저장 결과 노출

## 11. 세부 개발 순서

### Step 1. 프롬프트 자산 디렉토리 생성

할 일:

- `assets/prompts/chat/` 생성
- 현재 코드 상수 내용을 각 파일로 이동할 초안 작성

완료 기준:

- 프롬프트 파일 구조가 생기고 현재 프롬프트 문구가 모두 파일로 존재

### Step 2. Prompt Loader 구현

할 일:

- 설정에서 프롬프트 루트 경로 읽기
- 상대 경로로 텍스트 파일을 여는 로더 구현
- 파일 누락 시 예외 처리

완료 기준:

- 서비스 코드가 문자열 상수 대신 로더 호출로 프롬프트를 읽을 수 있음

### Step 3. Prompt Registry 정의

할 일:

- BO에서 노출할 prompt 파일 목록 정의
- key/title/description/path 구조 확정
- 임의 파일 접근이 불가능한 구조로 고정

완료 기준:

- 어떤 prompt를 운영에서 수정 가능한지 명시됨

### Step 4. prompt_builder 적용

할 일:

- `SYSTEM_PROMPT` import 제거
- `_SOURCES_INSTRUCTION`, `_QA_INSTRUCTION`, `_NO_SOURCES_GUARD` 상수 제거
- 파일 기반 로드 방식으로 치환

완료 기준:

- 현재 single-shot 메시지 조립이 파일 기반 프롬프트로 정상 동작

### Step 5. history_service 적용

할 일:

- 시스템 프롬프트 import 제거
- 초기 히스토리에서도 Prompt Loader 사용

완료 기준:

- 시스템 프롬프트 사용 경로가 한 군데도 코드 상수에 직접 의존하지 않음

### Step 6. BO Prompt 관리 페이지 구현 설계

할 일:

- 목록 화면/수정 화면 구조 정의
- 저장 동선 정의
- sidebar 메뉴 추가 위치 정의
- Django message 기반 저장 피드백 방식 정의

완료 기준:

- BO Prompt 관리 UI가 어떤 구조로 동작할지 명확해짐

### Step 7. 설정/문서 정리

할 일:

- `PROMPTS_DIR` 설정 추가
- README 수정
- 운영 시 외부 경로 override 방식 문서화
- BO Prompt 관리 사용 방식 문서화

완료 기준:

- 개발/운영 모두 프롬프트 위치를 설명 가능

### Step 8. 회귀 점검

할 일:

- 기존 single-shot 질문셋으로 응답 비교
- no-source 응답 확인
- 시스템 프롬프트 적용 여부 확인
- BO에서 prompt 수정 후 반영 흐름 확인

완료 기준:

- 기능 변화 없이 프롬프트 관리 방식만 분리되었음을 확인

## 12. 예상 위험 요소

### 10-1. 파일 누락

위험:

- 배포 환경에 프롬프트 파일이 누락되면 서비스가 실패할 수 있음

대응:

- 명시적 예외 처리
- README 및 배포 체크리스트에 프롬프트 파일 포함 확인 추가

### 10-2. repo 내부 경로만 사용

위험:

- 프롬프트를 파일로 뺐지만 여전히 이미지/배포물 내부라 재배포 없는 수정이 어려울 수 있음

대응:

- `PROMPTS_DIR` 환경변수 설계로 외부 경로 override 지원

### 10-3. 프롬프트 파일 난립

위험:

- 역할 구분 없이 파일이 늘어나면 관리 포인트가 오히려 증가

대응:

- 현재 Phase에서는 `chat/` 범위만 실제 사용
- 역할 기준 디렉토리 구조를 먼저 고정

### 12-4. BO를 통한 임의 파일 수정 위험

위험:

- 잘못 설계하면 운영 UI가 서버 파일 편집기로 변질될 수 있음

대응:

- registry 기반 allow-list
- 사용자 입력으로 path를 받지 않음
- 등록된 key로만 파일 접근

### 12-5. 운영 중 잘못된 prompt 저장

위험:

- 빈 값 저장 또는 부적절한 prompt 저장으로 품질 저하 가능

대응:

- 최소 길이/빈 값 검증 검토
- 저장 전 확인 문구 제공
- 필요 시 추후 rollback 기능 추가

## 13. 완료 정의

Phase 1은 아래 조건을 만족하면 완료로 본다.

- 시스템 프롬프트와 single-shot 관련 보조 프롬프트가 파일로 분리되었다.
- `Prompt Loader`를 통해 프롬프트를 읽는다.
- `prompt_builder.py`와 `history_service.py`가 코드 상수에 직접 의존하지 않는다.
- `PROMPTS_DIR` 설정으로 프롬프트 루트 경로를 제어할 수 있다.
- BO에서 편집 가능한 prompt 목록과 수정 화면 구조가 정의되었다.
- BO가 허용된 prompt만 수정할 수 있는 방식으로 설계되었다.
- 기존 single-shot 답변 동작이 유지된다.
- README에 프롬프트 관리 위치와 운영 방식이 정리된다.

## 14. Phase 1 완료 후 다음 단계 연결

Phase 1이 끝나면 다음 단계인 `LangGraph 적용`에서 아래 이점을 바로 가져갈 수 있다.

- node별 프롬프트 파일 분리 준비 완료
- single-shot node 래핑 시 prompt import 구조 단순화
- 이후 workflow/agent prompt 추가 시 동일한 loader 패턴 재사용 가능
- 운영 측면에서 prompt 수정 진입점이 BO에 확보됨

즉 Phase 1은 단순한 문자열 분리가 아니라, `LangGraph 기반 구조 전환을 위한 프롬프트 계층 정리 단계`로 본다.
