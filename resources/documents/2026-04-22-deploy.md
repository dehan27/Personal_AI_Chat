# 2026-04-22 개발 로그 — 프로덕션 배포 파이프라인

## 배경
`v0.2.0`까지는 로컬 개발 전용 컴포즈(`runserver`)만 있었음. 운영 배포를 위해 **GitHub Actions → GHCR → 운영 호스트** 자동화 파이프라인이 필요.

목표: 태그 하나만 푸시하면 이미지 빌드·퍼블리시·원격 재기동까지 전부 자동. 런타임 시크릿은 운영 호스트에만 두고 GitHub에는 배포 접속 정보만 보관.

## 1. 프로덕션 런너 전환

**`Dockerfile.prod` 신설**
- 베이스: `python:3.11-slim`
- 빌드 시: `apt-get install build-essential libpq-dev` → pip install → 빌드 도구 purge
- `collectstatic --noinput`을 빌드 단계에서 실행 (런타임 I/O 절약)
- 비루트 사용자 `app` (UID 1000)로 실행 권한 이관
- CMD: `migrate --noinput && gunicorn --workers 3 --threads 4 --worker-class gthread --timeout 120`

**`requirements.txt`**
- `gunicorn>=22`, `whitenoise>=6.6` 추가

**`AI_Chat/settings.py` 환경변수화**
- `DJANGO_DEBUG` — 기본 `False` (운영 안전). 로컬은 `.env`에 `True` 명시
- `DJANGO_ALLOWED_HOSTS`, `DJANGO_CSRF_TRUSTED_ORIGINS` — 쉼표 구분 env 리스트 헬퍼
- `WhiteNoise` 미들웨어 + `CompressedManifestStaticFilesStorage` 등록
- `DEBUG=False` 시 `SECURE_PROXY_SSL_HEADER`, `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE` 자동 활성화

**`.dockerignore` 신설**
- `.env*`, `.git/`, `resources/`, `staticfiles/`, `__pycache__/` 등 제외
- 없을 때: `.env`가 이미지에 박혀 레지스트리로 유출될 위험 → 치명적
- 적용 후 이미지 `/app` 크기 ~6 MB

## 2. 컴포즈 (`docker-compose.prod.yml`)
- `web`: `ghcr.io/sean-59/personal_ai_chat:${APP_IMAGE_TAG:-latest}`
- `db`: `pgvector/pgvector:pg17`
- 포트·볼륨 매핑은 전부 env 로 파라미터화 (실값은 운영 호스트 `.env` 에만)
- 볼륨: `./resources` (업로드 파일), `./pgdata` (DB) — 배포 디렉토리 하위에 영속화
- `restart: unless-stopped` 로 재부팅·크래시 복구
- `db` healthcheck(`pg_isready`) 통과 후 web 기동

**`env.prod.example.txt`**
- 운영 전용 템플릿. 모든 값은 플레이스홀더 (실값은 운영 호스트 `.env`에만 둠)
- `DJANGO_DEBUG=False`, `APP_IMAGE_TAG=latest` 기본값 제공

---

## 3. GitHub Actions 워크플로우

**`.github/workflows/deploy.yml`**
- 트리거: `push` 태그 `v*` + `workflow_dispatch`(수동)
- **build-and-push 잡**
  - `docker/setup-buildx-action` + `build-push-action`
  - `Dockerfile.prod` 빌드 → `ghcr.io/sean-59/personal_ai_chat:<태그>` 와 `:latest` 둘 다 push
  - GitHub Actions 캐시(`type=gha`)로 재빌드 가속
  - `GITHUB_TOKEN` + `packages:write` 권한으로 GHCR 인증
- **deploy 잡**
  - 5개 Secrets 사용: `SSH_PRIVATE_KEY`, `DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_PORT`, `DEPLOY_PATH`
  - 운영 호스트 SSH → `$DEPLOY_PATH/.env`의 `APP_IMAGE_TAG` 값 `sed`로 치환
  - `docker compose pull && up -d && image prune -f`

## 4. 초기 셋업 요구사항 (1회)

### GitHub
- `Settings → Actions → General → Workflow permissions` = **Read and write** (GHCR push 용)
- Repository Secrets 등록 5개: `SSH_PRIVATE_KEY`, `DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_PORT`, `DEPLOY_PATH`
- 첫 배포 후 **Packages** 탭에서 GHCR 가시성을 원하는 대로 설정 (Public 이면 호스트에서 로그인 없이 pull 가능)

### 운영 호스트
- 배포 디렉토리 생성 (`DEPLOY_PATH` 로 등록한 경로)
- `docker-compose.prod.yml` → `docker-compose.yml`로 이름 바꿔 배치
- `.env` 작성 (`env.prod.example.txt` 복사 후 값 채움)
- 배포용 SSH 공개키를 `~/.ssh/authorized_keys` 에 등록
- 홈·`.ssh` 권한: `755 ~` / `700 ~/.ssh` / `600 ~/.ssh/authorized_keys`
- 리버스 프록시가 공개 도메인(443) → 웹 컨테이너 매핑 포트로 포워딩 (SSL 종단 포함)

## 5. 배포 흐름

```
git tag v0.3.0 && git push origin v0.3.0
  ↓ (on: push tags v*)
GitHub Actions
  ├── build-and-push: Dockerfile.prod 빌드 → GHCR push
  └── deploy: SSH → APP_IMAGE_TAG 갱신 → compose pull/up
        ↓
운영 호스트 컨테이너 재기동
  ├── db: pgdata 볼륨 유지
  └── web: migrate 자동 실행 (pgvector + pg_trgm 확장 포함)
        ↓
운영 도메인으로 접속 가능
```

첫 배포 시 `migrate`가 `chat/0001_vector_extension`, `files/0003_trigram_search`에서 두 extension을 자동 설치하므로 수동 SQL 불필요.
